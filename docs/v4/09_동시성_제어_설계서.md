## 1. 문제 정의

### 1-1. 좌석 Hold Race Condition

```
T=0ms  사용자 A: POST /api/events/42/seats/101/hold  ──┐
T=1ms  사용자 B: POST /api/events/42/seats/101/hold  ──┤──▶ 동시 도달
T=2ms  사용자 C: POST /api/events/42/seats/101/hold  ──┘

기대: A만 Hold 성공, B·C는 즉시 409
위험: 3명 모두 ACTIVE_BOOKING에 seat_id=101이 없음을 확인하고 모두 Hold 진행
결과(제어 없을 때): 3명 모두 Hold 성공 → 중복 예매 발생
```

### 1-2. 선착순 쿠폰 발급 Race Condition

```
T=0ms  500명 동시: POST /api/coupons/3/issue

쿠폰 잔여 수량: 10개
기대: 10명만 발급 성공, 나머지 490명은 409
위험: 500명이 동시에 remaining_quantity > 0 확인 후 모두 발급 시도
결과(제어 없을 때): 10개 이상 발급 → 재고 초과
```

### 1-3. 왜 단순 체크로는 부족한가?

```
Thread A: SELECT hold:{42}:{101} → 없음  (읽기)
Thread B: SELECT hold:{42}:{101} → 없음  (읽기, A가 아직 SET 전)
Thread A: SET hold:{42}:{101} = userId1  (쓰기)
Thread B: SET hold:{42}:{101} = userId2  (쓰기, 덮어씀)
→ Lost Update 문제 — 두 사람 모두 Hold 성공으로 인식
```

## 2. 동시성 제어 전략 비교

| 전략 | 메커니즘 | 장점 | 단점 | TicketFlow 적용 |
|------|----------|------|------|----------------|
| **낙관적 락** | `@Version` + 충돌 시 재시도 | 충돌 적을 때 성능 우수 | 재시도 로직 복잡, 충돌 많으면 성능 저하 | USER_COUPON 사용 처리에만 부분 적용 |
| **비관적 락** | `SELECT ... FOR UPDATE` | 구현 단순, 정합성 확실 | 락 대기 지연, 단일 DB 의존 | 주문 취소·수동 확정 등 DB 트랜잭션 범위 내 처리 |
| **Redis Lettuce SETNX** | SETNX + TTL + Lua Script | 비즈니스 로직 전체 보호, Stateless | 네트워크 비용, 구현 복잡 | 좌석 Hold ✅ 필수 구현 |
| **Redis DECR + Lua** | DECR 원자적 연산 | 분산락 없이 원자적 수량 차감 | 음수 방어 Lua Script 필요 | 쿠폰 발급 ✅ 필수 구현 |
| **Redisson** | Redis SetNX + TTL + Watchdog | AOP 구조, 편의성 | Redis 의존성, 복잡도 | 도전 기능(Lettuce 완성 후 리팩토링) |
| **DB UNIQUE 제약** | ACTIVE_BOOKING PK(seat_id) | DB 레벨 최종 방어 | 에러 처리 필요 | 2차 방어선으로 병행 적용 |

**락 획득 순서 (데드락 방지):**
```
① 좌석 Hold 락   (Redis SETNX: lock:seat:{eventId}:{seatId})
② 쿠폰 사용 락   (Redis SETNX: lock:user-coupon-use:{userCouponId})
③ DB 트랜잭션    (MySQL InnoDB)
```
> 항상 ①→②→③ 순서 고정. 해제는 역순(③→②→①). 순서가 다르면 Thread A↔B 데드락 발생.

## 3. 시나리오 매트릭스

### 시나리오 A: 좌석 Hold — Lettuce SETNX 분산락 (필수)

| 단계 | 사용자 A (Thread1) | 사용자 B (Thread2) | Redis 상태 |
|------|------------------|------------------|-----------|
| T=0 | `SETNX lock:seat:42:101 {uuidA} EX 3` → 성공 | `SETNX lock:seat:42:101 {uuidB} EX 3` → 실패 | lock 키 SET |
| T=1 | ACTIVE_BOOKING에 seat_id=101 없음 확인 | 즉시 409 SEAT_LOCK_FAILED 반환 | - |
| T=2 | `SET hold:42:101 {userId1} EX 300` | - | hold 키 SET |
| T=3 | `SET holdToken:{uuidA} "42:101:{userId1}" EX 300` | - | holdToken 키 SET |
| T=4 | `INCR user-hold-count:{userId1}` | - | count INCR |
| T=5 | Lua Script: 본인 락만 원자적 삭제 | - | lock 키 DEL |
| T=6 | 200 OK `{ holdToken, expiresAt }` 반환 | - | - |

**결과**: A만 Hold 성공, B는 DB에 전혀 도달하지 않음 ✅

**코드 예시:**
```java
// HoldLockService.java — Lettuce SETNX 기반
public HoldResponse holdSeat(Long eventId, Long seatId, Long userId) {
    String lockKey = "lock:seat:" + eventId + ":" + seatId;
    String lockValue = UUID.randomUUID().toString();

    // 1. 분산락 획득 시도 (Fail Fast — waitTime=0)
    Boolean acquired = redisTemplate.opsForValue()
        .setIfAbsent(lockKey, lockValue, Duration.ofSeconds(3));

    if (!Boolean.TRUE.equals(acquired)) {
        throw new ConflictException("SEAT_LOCK_FAILED", "다른 사용자가 처리 중입니다.");
    }

    try {
        // 2. CONFIRMED 여부 확인 (ACTIVE_BOOKING)
        if (activeBookingRepository.existsBySeatId(seatId)) {
            throw new ConflictException("SEAT_ALREADY_CONFIRMED", "이미 예매된 좌석입니다.");
        }
        // 3. Hold 여부 확인
        if (redisTemplate.hasKey("hold:" + eventId + ":" + seatId)) {
            throw new ConflictException("SEAT_ALREADY_HELD", "이미 선점된 좌석입니다.");
        }
        // 4. 어뷰징 체크 (4석 제한)
        Integer count = (Integer) redisTemplate.opsForValue()
            .get("user-hold-count:" + userId);
        if (count != null && count >= 4) {
            throw new BadRequestException("HOLD_LIMIT_EXCEEDED", "최대 4석까지만 선택할 수 있습니다.");
        }
        // 5. Hold 키·holdToken 키·count 저장
        String holdToken = UUID.randomUUID().toString();
        redisTemplate.opsForValue().set("hold:" + eventId + ":" + seatId,
            String.valueOf(userId), Duration.ofSeconds(300));
        redisTemplate.opsForValue().set("holdToken:" + holdToken,
            eventId + ":" + seatId + ":" + userId, Duration.ofSeconds(300));
        redisTemplate.opsForValue().increment("user-hold-count:" + userId);
        redisTemplate.expire("user-hold-count:" + userId, Duration.ofSeconds(300));

        return new HoldResponse(holdToken, LocalDateTime.now().plusMinutes(5));
    } finally {
        // 6. 본인 락만 원자적 삭제 (Lua Script)
        String luaScript = """
            if redis.call('get', KEYS[1]) == ARGV[1] then
                return redis.call('del', KEYS[1])
            else
                return 0
            end
            """;
        redisTemplate.execute(new DefaultRedisScript<>(luaScript, Long.class),
            List.of(lockKey), lockValue);
    }
}
```

### 시나리오 B: 좌석 Hold — Redisson AOP 리팩토링 (도전)

> Lettuce 기반 구현이 완성되고 테스트가 통과한 이후 진행.

| 단계 | 사용자 A (Thread1) | 사용자 B (Thread2) | Redis 상태 |
|------|------------------|------------------|-----------|
| T=0 | `tryLock("lock:seat:42:101", wait=0, lease=3s)` | `tryLock(...)` 동시 호출 | - |
| T=1 | 락 획득 성공 | 락 획득 실패 → 즉시 409 반환 | key SET, TTL=3s |
| T=2 | Hold 로직 실행 | - | hold 키 SET |
| T=3 | `unlock()` — finally 블록 | - | lock 키 DEL |
| T=4 | 200 OK 반환 | 409 이미 반환됨 | - |

**코드 예시 (AOP 패턴):**
```java
// @RedisLock 커스텀 어노테이션
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface RedisLock {
    String key();           // SpEL 표현식
    long leaseTime() default 3;
    TimeUnit timeUnit() default TimeUnit.SECONDS;
}

// RedisLockAspect.java
@Aspect
@Component
@RequiredArgsConstructor
public class RedisLockAspect {
    private final RedissonClient redissonClient;

    @Around("@annotation(redisLock)")
    public Object around(ProceedingJoinPoint pjp, RedisLock redisLock) throws Throwable {
        String lockKey = SpELParser.parse(redisLock.key(), pjp); // SpEL 키 파싱
        RLock lock = redissonClient.getLock(lockKey);

        boolean acquired = false;
        try {
            acquired = lock.tryLock(0, redisLock.leaseTime(), redisLock.timeUnit());
            if (!acquired) throw new ConflictException("SEAT_LOCK_FAILED", "다른 사용자가 처리 중입니다.");
            return pjp.proceed();
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RuntimeException("락 획득 중 인터럽트 발생", e);
        } finally {
            if (acquired && lock.isHeldByCurrentThread()) lock.unlock();
        }
    }
}

// HoldService.java — 어노테이션만 붙이면 락 적용
@RedisLock(key = "'lock:seat:' + #eventId + ':' + #seatId")
public HoldResponse holdSeat(Long eventId, Long seatId, Long userId) {
    // 락 로직이 AOP로 분리되어 비즈니스 로직만 남음
    ...
}
```

### 시나리오 C: 쿠폰 발급 — Redis DECR + Lua Script (필수)

> DECR 자체가 원자적 연산이므로 추가 분산락 불필요 (L-01 대응).

| 단계 | 사용자 A | 사용자 B (500명째) | Redis coupon:stock:3 |
|------|---------|-----------------|---------------------|
| T=0 | Lua DECR → 9 반환 | Lua DECR → 0 반환 | 10 |
| T=1 | 0 이상 → 발급 진행 | 0 이상 → 발급 진행 | 9 |
| ... | ... | ... | ... |
| T=10 | Lua DECR → 0 반환 | Lua DECR → **-1** 반환 | 0 |
| T=11 | 마지막 발급 성공 | -1 < 0 → INCR 원상복구 → 409 | 0 (복구됨) |

**Lua Script (음수 방어):**
```lua
-- couponDecr.lua
local val = redis.call('DECR', KEYS[1])
if val < 0 then
    redis.call('INCR', KEYS[1])   -- 원상복구
    return -1                      -- 소진 신호
end
return val
```

**처리 흐름:**
```java
// CouponService.java
public void issueCoupon(Long couponId, Long userId) {
    // 1. 중복 발급 조기 차단 (락 없이 빠른 실패)
    if (userCouponRepository.existsByUserIdAndCouponId(userId, couponId)) {
        throw new ConflictException("COUPON_ALREADY_ISSUED", "이미 발급받은 쿠폰입니다.");
    }
    // 2. Redis DECR + Lua (원자적 수량 차감)
    Long result = redisTemplate.execute(couponDecrScript,
        List.of("coupon:stock:" + couponId));
    if (result == null || result < 0) {
        throw new ConflictException("COUPON_EXHAUSTED", "쿠폰이 모두 소진되었습니다.");
    }
    // 3. DB INSERT + MySQL remaining_quantity 즉시 동기화 (m-06)
    userCouponRepository.save(UserCoupon.of(userId, couponId));
    couponRepository.decrementRemainingQuantity(couponId);  // UPDATE ... SET remaining_quantity - 1
}
```

**DB fallback (Redis 연결 자체 실패 시):**
```java
// Redis 연결 실패 시 catch → DB 경로로 전환
@Transactional
public void issueCouponFallback(Long couponId, Long userId) {
    Coupon coupon = couponRepository.findByIdForUpdate(couponId); // SELECT ... FOR UPDATE
    if (coupon.getRemainingQuantity() <= 0) {
        throw new ConflictException("COUPON_EXHAUSTED", "쿠폰이 모두 소진되었습니다.");
    }
    coupon.decrementQuantity();
    userCouponRepository.save(UserCoupon.of(userId, couponId));
}
```

### 시나리오 D: 웹훅 수신 — Hold TTL 만료 Race Condition (L-02)

**상황**: 주문 생성 시점에 Hold 잔여 TTL이 30초밖에 없는 경우

| 단계 | 서버 | 상태 |
|------|------|------|
| T=0 | 사용자 A가 좌석 Hold 성공 | hold TTL = 300초 |
| T=4분30초 | 사용자 A가 `POST /api/orders` 요청 | hold TTL = 30초 남음 |
| T=4분31초 | 주문 생성 성공 → **모든 hold 키 TTL +5분 연장** (L-02) | hold TTL = 300초로 재설정 |
| T=4분32초 | Mock PG 결제 요청 전송 | - |
| T=4분40초 | PG 결제 완료 웹훅 수신 | hold TTL = 292초 (충분) |
| T=4분41초 | Hold 유효성 검증 성공 → 예매 확정 | hold 키 DEL |

**TTL 연장 구현:**
```java
// OrderService.java — 주문 생성 성공 직후
private void extendHoldTtl(List<String> holdTokens) {
    for (String holdToken : holdTokens) {
        String tokenValue = (String) redisTemplate.opsForValue().get("holdToken:" + holdToken);
        if (tokenValue != null) {
            String[] parts = tokenValue.split(":");
            String holdKey = "hold:" + parts[0] + ":" + parts[1];
            redisTemplate.expire(holdKey, Duration.ofSeconds(300));
            redisTemplate.expire("holdToken:" + holdToken, Duration.ofSeconds(300));
        }
    }
}
```

### 시나리오 E: Hold TTL 만료 후 결제 완료 웹훅 수신

**상황**: L-02 TTL 연장에도 불구하고 네트워크 장애 등으로 웹훅이 5분 이후 도착

| 단계 | 서버 | 상태 |
|------|------|------|
| T=0 | 주문 생성 + Hold TTL 연장 | hold TTL = 300초 |
| T=5분 | Hold TTL 자연 만료 | hold 키 없음 |
| T=5분30초 | PG 웹훅 수신 | BOOKING.status = PENDING |
| T=5분31초 | 웹훅 내 hold 키 TTL 검증 → **만료 확인** | - |
| T=5분32초 | 예매 실패 처리 (ORDER/BOOKING → FAILED) | - |
| T=5분33초 | PG 결제 취소 요청 (환불) | - |

**웹훅 검증 로직:**
```java
// WebhookService.java
private void validateHoldTtl(List<Booking> bookings) {
    for (Booking booking : bookings) {
        String holdKey = "hold:" + booking.getEventId() + ":" + booking.getSeatId();
        if (!Boolean.TRUE.equals(redisTemplate.hasKey(holdKey))) {
            // Hold 만료 → 예매 실패 처리
            failOrder(booking.getOrderId());
            requestPgRefund(booking.getOrderId());
            throw new HoldExpiredException("점유 시간이 만료되어 예매가 취소되었습니다.");
        }
    }
}
```

### 시나리오 F: 주문 취소 — SELECT FOR UPDATE 비관적 락 (C-03)

**상황**: 사용자의 취소 트랜잭션과 PG 웹훅 확정 트랜잭션이 동시 도달

| 단계 | 취소 트랜잭션 | 웹훅 확정 트랜잭션 | DB 상태 |
|------|------------|-----------------|---------|
| T=0 | 트랜잭션 시작 | 트랜잭션 시작 | ORDER=CONFIRMED |
| T=1 | `SELECT * FROM booking WHERE order_id=? FOR UPDATE` → 락 획득 | `SELECT * FROM booking WHERE order_id=? FOR UPDATE` → **대기** | 락 보유: 취소 TX |
| T=2 | `SELECT * FROM order WHERE order_id=? FOR UPDATE` | (대기 중) | - |
| T=3 | `SELECT * FROM active_booking WHERE seat_id IN(?) FOR UPDATE` | (대기 중) | - |
| T=4 | `SELECT * FROM user_coupon WHERE user_coupon_id=? FOR UPDATE` | (대기 중) | - |
| T=5 | ACTIVE_BOOKING DELETE, BOOKING→CANCELLED, ORDER→CANCELLED, USER_COUPON→ISSUED | (대기 중) | 모두 CANCELLED |
| T=6 | COMMIT, 락 해제 | 락 획득 → ORDER.status=CANCELLED 확인 → 롤백 | CANCELLED |
| T=7 | - | 409 반환 (이미 취소됨) | - |

**결과**: 취소와 웹훅 확정의 충돌 없이 정합성 보장 ✅

### 시나리오 G: ACTIVE_BOOKING PK 제약 — 최후 방어선

**상황**: Redis 락 버그 또는 장애로 동시에 두 트랜잭션이 CONFIRMED 처리 시도

| 단계 | Thread A | Thread B | DB |
|------|---------|---------|-----|
| T=0 | DB 트랜잭션 진입 | DB 트랜잭션 진입 | seat_id=101 ACTIVE_BOOKING 없음 |
| T=1 | `INSERT INTO active_booking (seat_id, booking_id) VALUES (101, 201)` → 성공 | `INSERT INTO active_booking (seat_id, booking_id) VALUES (101, 202)` → **PK 중복 에러** | seat_id=101 존재 |
| T=2 | COMMIT | 즉시 ROLLBACK | seat_id=101 고정 |

```sql
-- ACTIVE_BOOKING DDL — PK가 중복 확정을 DB 레벨에서 원천 차단
CREATE TABLE active_booking (
    seat_id    BIGINT PRIMARY KEY,        -- 좌석당 1건만 허용
    booking_id BIGINT UNIQUE NOT NULL,
    FOREIGN KEY (seat_id)    REFERENCES seat(seat_id),
    FOREIGN KEY (booking_id) REFERENCES booking(booking_id)
);
```

> Redis 분산락이 1차 방어선, ACTIVE_BOOKING PK 제약이 2차 방어선.
> 락 구현 버그나 Redis 장애 상황에서도 중복 확정을 막는다.

### 시나리오 H: 사용자 동일 좌석 중복 Hold 시도 (멱등성)

**상황**: 사용자가 네트워크 지연으로 동일 좌석에 Hold 요청을 2번 전송

| 단계 | 요청 1 | 요청 2 | Redis |
|------|--------|--------|-------|
| T=0 | `SETNX lock:seat:42:101 {uuid1} EX 3` → 성공 | `SETNX lock:seat:42:101 {uuid2} EX 3` → **실패** | lock 키 존재 |
| T=1 | Hold 처리 진행 | 즉시 409 SEAT_LOCK_FAILED 반환 | - |
| T=2 | hold:42:101 = userId1 SET | - | hold 키 SET |
| T=3 | COMMIT, 락 해제 | - | lock 키 DEL |
| T=4 | 200 OK | 409 이미 반환됨 | - |

**락 해제 후 재시도 시 추가 방어:**
```java
// 5번 단계: hold 키 존재 확인으로 2차 차단
if (redisTemplate.hasKey("hold:" + eventId + ":" + seatId)) {
    throw new ConflictException("SEAT_ALREADY_HELD", "이미 선점된 좌석입니다.");
}
```

### 시나리오 I: leaseTime 만료 중 처리 지연

**상황**: 분산락 leaseTime(3초) 내에 비즈니스 로직이 완료되지 않을 경우

| 단계 | Thread A | Thread B | Redis | DB |
|------|---------|---------|-------|----|
| T=0 | `tryLock(wait=0, lease=3s)` 성공 | `tryLock` 실패 → 409 | lock TTL=3s | AVAILABLE |
| T=3s | **leaseTime 만료** — lock 키 자동 삭제 | - | lock 키 없음 | 미완료 |
| T=3.1s | (비즈니스 로직 아직 진행 중) | 새 요청: `tryLock` 성공 가능 | - | - |
| T=3.2s | - | Hold 키 존재 확인 → `SEAT_ALREADY_HELD` → 409 | hold 키 존재 | - |
| T=4s | 정상 완료 | 이미 409 반환 | - | HOLD 완료 |

**결과**: leaseTime 만료됐어도 hold 키 존재 여부 체크가 2차 방어선으로 동작 ✅

> **왜 Watchdog(자동 연장)을 사용하지 않는가?**
> ```
> tryLock() → leaseTime 고정, 자동 연장 없음 (본 프로젝트 선택)
> lock()    → Watchdog 활성화, 30초마다 자동 연장
>
> ∴ tryLock 사용 시 leaseTime은 실제 처리 시간보다 충분히 길어야 한다.
>   현재 설정 3초는 DB 트랜잭션 통상 100ms~500ms 기준 여유 6배.
>   외부 API(Mock PG) 호출이 추가되면 leaseTime 재검토 필요.
> ```

## 4. 좌석 상태 전이와 락 적용 범위

```
[좌석 상태 전이와 동시성 제어 적용 범위]

AVAILABLE ──────────────────────────▶ ON_HOLD ──────────────────▶ CONFIRMED
           ← Redis SETNX 분산락 적용 →          ← 웹훅 DB 트랜잭션 →
           ← (ACTIVE_BOOKING PK 2차 방어선) ──────────────────────────────▶

락 저장소별 역할:
  Redis hold 키:         ON_HOLD 상태 관리 (TTL 300초)
  Redis holdToken 키:    주문 생성 시 역조회용 (TTL 300초, C-01)
  Redis lock 키:         Hold 획득 경쟁 구간 보호 (TTL 3초)
  Redis user-hold-count: 4석 제한 어뷰징 방지 (TTL 300초, M-02)
  MySQL ACTIVE_BOOKING:  CONFIRMED 상태 관리 (PK 중복 방어)
  MySQL BOOKING:         모든 상태 이력 보존

동시성 제어 필요 여부:
  AVAILABLE → ON_HOLD    : N명 동시 시도 가능 → Redis SETNX 분산락 ✅ 필수
  ON_HOLD → CONFIRMED    : 웹훅 단일 경로, 멱등성 처리 → DB 트랜잭션 + PK 방어 ✅
  CONFIRMED → CANCELLED  : 취소 트랜잭션 + 웹훅 충돌 가능 → SELECT FOR UPDATE ✅
  ON_HOLD → AVAILABLE    : TTL 자연 만료 또는 수동 해제 → 락 불필요
```

## 5. 동시성 테스트 시나리오

### 5-1. 좌석 Hold 동시성 테스트

```java
@SpringBootTest
class HoldConcurrencyTest {

    @Test
    @DisplayName("동시에 100명이 동일 좌석 Hold 시도 시 1명만 성공해야 한다")
    void concurrentHoldTest() throws InterruptedException {
        // given
        int threadCount = 100;
        ExecutorService executor = Executors.newFixedThreadPool(threadCount);
        CountDownLatch latch = new CountDownLatch(threadCount);
        AtomicInteger successCount = new AtomicInteger(0);
        AtomicInteger failCount = new AtomicInteger(0);

        Long eventId = 42L;
        Long seatId = 101L;

        // when
        for (int i = 0; i < threadCount; i++) {
            final long userId = i + 1;
            executor.submit(() -> {
                try {
                    holdService.holdSeat(eventId, seatId, userId);
                    successCount.incrementAndGet();
                } catch (ConflictException e) {
                    failCount.incrementAndGet();
                } finally {
                    latch.countDown();
                }
            });
        }
        latch.await(10, TimeUnit.SECONDS);

        // then
        assertThat(successCount.get()).isEqualTo(1);
        assertThat(failCount.get()).isEqualTo(99);
        assertThat(redisTemplate.hasKey("hold:" + eventId + ":" + seatId)).isTrue();
    }
}
```

### 5-2. 쿠폰 발급 동시성 테스트

```java
@Test
@DisplayName("동시에 500명 요청 시 쿠폰 잔여 수량(10개)만큼만 발급되어야 한다")
void concurrentCouponIssueTest() throws InterruptedException {
    // given
    int threadCount = 500;
    int couponStock = 10;
    // Redis에 초기 재고 설정
    redisTemplate.opsForValue().set("coupon:stock:3", String.valueOf(couponStock));

    ExecutorService executor = Executors.newFixedThreadPool(threadCount);
    CountDownLatch latch = new CountDownLatch(threadCount);
    AtomicInteger successCount = new AtomicInteger(0);

    // when
    for (int i = 0; i < threadCount; i++) {
        final long userId = i + 1;
        executor.submit(() -> {
            try {
                couponService.issueCoupon(3L, userId);
                successCount.incrementAndGet();
            } catch (ConflictException ignored) {
            } finally {
                latch.countDown();
            }
        });
    }
    latch.await(30, TimeUnit.SECONDS);

    // then
    assertThat(successCount.get()).isEqualTo(couponStock);
    long issuedCount = userCouponRepository.countByCouponId(3L);
    assertThat(issuedCount).isEqualTo(couponStock); // DB 발급 수도 정확히 10건
}
```

### 5-3. 구현 순서 (발제 요구사항)

```
[1단계] 동시성 테스트 코드 작성 (실패 확인)
  └ 위 테스트 코드 먼저 작성
  └ 이 시점에 테스트는 반드시 실패해야 정상

        ↓

[2단계] Lettuce SETNX 분산락 구현 (필수)
  └ DistributedLockProvider 인터페이스 정의 (Feature Flag 추상화)
  └ LettuceDistributedLock 구현체 작성
  └ LockConfig — lock.provider=lettuce (matchIfMissing=true) 등록
  └ HoldLockFacade는 DistributedLockProvider 인터페이스만 주입
  └ 테스트 재실행 → 통과 확인

        ↓ 도전 구현

[3단계] Redisson 구현 + Feature Flag 전환
  └ RedissonDistributedLock 구현체 추가
  └ LockConfig에 lock.provider=redisson Bean 등록
  └ @RedisLock 커스텀 어노테이션 + @Aspect 구조 추가 (선택)
  └ 동일 테스트 코드로 Redisson 검증 (코드 변경 없이 yml만 변경)
  └ Lettuce 대비 Redisson 선택 근거 README에 정리

[전환 방법]
  application.yml: lock.provider: lettuce → redisson
  문제 발생 시:    lock.provider: redisson → lettuce  (즉시 롤백)
```

> **캐시 전략과 대칭 구조:**
> ```yaml
> cache:
>   provider: caffeine   # caffeine | redis    (ADR-006)
> lock:
>   provider: lettuce    # lettuce  | redisson (ADR-002)
> ```

## 6. 모니터링 포인트

| 항목 | 확인 방법 | 임계값 / 조치 |
|------|-----------|-------------|
| Hold 락 획득 실패율 | 409 SEAT_LOCK_FAILED 로그 비율 | 높으면 UX 안내 문구 강화 |
| Hold TTL 만료 후 웹훅 수신 | HoldExpiredException 로그 | 빈번 시 PG 웹훅 응답 속도 점검 |
| Redisson leaseTime 초과 | leaseTime 내 트랜잭션 완료 여부 모니터링 | 초과 시 leaseTime 값 증가 (현재 3s) |
| 쿠폰 Redis ↔ MySQL 불일치 | `coupon:stock:{id}` vs `remaining_quantity` 주기적 비교 | 차이 발생 시 MySQL 기준으로 Redis 재초기화 |
| ACTIVE_BOOKING PK 충돌 | DataIntegrityViolationException 로그 | 발생 = 분산락 버그 → 즉시 조사 |
| DB 락 대기 타임아웃 | `LockTimeoutException` 로그 | 빈번 시 `lock.timeout` 값 조정 |

### 설정값 기준표

| 설정 | 현재값 | 이유 |
|------|--------|------|
| 좌석 Hold TTL | 300초 (5분) | 결제 완료까지 여유 시간 |
| 분산락 leaseTime | 3초 | 통상 DB 트랜잭션 500ms 기준 여유 6배 |
| 분산락 waitTime | 0초 | Fail Fast — 즉시 실패가 티켓팅 UX에 유리 |
| `javax.persistence.lock.timeout` | 3,000ms | 기본 50s → UX 개선 |
| 쿠폰 발급 Retry | 최대 3회 Backoff | "받느냐 못 받느냐" UX — 잠깐 기다려서라도 받는 게 유리 |
| user-hold-count TTL | 300초 | Hold TTL과 동기화, 만료 시 자동 정리 |
