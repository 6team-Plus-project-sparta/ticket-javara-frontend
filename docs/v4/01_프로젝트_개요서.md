## 2. 프로젝트 배경 및 목적

### 배경

온라인 티켓팅 시장은 순간적으로 수천~수만 명이 동시에 접속하는 **극단적 트래픽 집중** 환경이다.
인터파크, 멜론티켓, YES24 티켓 등 기존 서비스에서도 오픈 직후 서버 다운, 중복 예매, 재고 불일치 등의 문제가 반복적으로 발생한다.

### 목적

- 대규모 동시 요청에서도 **데이터 정합성을 보장**하는 티켓팅 백엔드 구현
- 캐싱 전략으로 **검색 성능을 최적화**하는 경험 습득
- 실제 운영 환경과 유사한 **CI/CD 파이프라인 및 클라우드 배포** 경험
- 백엔드 개발자로서 반드시 마주치는 **동시성·성능·운영** 문제를 직접 해결

## 3. 서비스 개요

**TicketFlow**는 공연(콘서트, 뮤지컬, 연극)과 스포츠(야구, 축구, 농구) 티켓을 온라인으로 예매할 수 있는 플랫폼이다.

### 핵심 가치 제안

| 가치 | 설명 |
|------|------|
| 공정한 선착순 | Redis 분산락으로 중복 예매 없는 정확한 선착순 보장 |
| 빠른 검색 | Caffeine 로컬 캐시 + Redis 캐시로 인기 검색어 즉시 응답 |
| 투명한 프로모션 | 선착순 할인 쿠폰 발급 시 정확한 수량 제한 |
| 빠른 CS 응답 | WebSocket 기반 실시간 CS 문의 채팅 |

## 4. 핵심 비즈니스 로직: 좌석 예매 플로우 ⭐

> 티켓팅 도메인 특성상 단순 `선택 → 결제 → 확정` 플로우는 치명적 결함이 있어 **좌석 임시 점유(Hold)** 개념을 도입한다.

### 4-1. 문제 정의

인기 콘서트 오픈 시 1만 명이 동시에 동일 좌석을 클릭한다고 가정하자.

- **Hold 없는 구조:** 1만 명이 모두 결제창에 진입 → PG사 결제 완료 → 이후 서버에서 "이미 예매된 좌석" 에러 → 9,999명 결제 취소 + 환불 처리
- **Hold 있는 구조:** 좌석 선택 시점에 1명만 점유 성공 → 나머지는 즉시 "선점됨" 안내 → 불필요한 결제 트래픽 0

### 4-2. 좌석 상태 머신

```
[AVAILABLE] ──좌석 선택──▶ [ON_HOLD] ──결제 완료──▶ [CONFIRMED]
                                │
                         TTL 만료(5분)
                                │
                                ▼
                          [AVAILABLE]  ← 자동 해제
```

| 상태 | 설명 | 저장소 |
|------|------|--------|
| `AVAILABLE` | 예매 가능 | ACTIVE_BOOKING 미존재 + Redis hold 키 미존재 (SEAT 테이블에 status 컬럼 없음, ERD v7.0 기준) |
| `ON_HOLD` | 임시 점유 (5분 TTL) | Redis (`hold:{eventId}:{seatId}` → userId) |
| `CONFIRMED` | 예매 확정 | MySQL ACTIVE_BOOKING 테이블에 해당 seat_id 행 존재 |
| `CANCELLED` | 취소 | ACTIVE_BOOKING 행 삭제 → AVAILABLE로 복귀, BOOKING.status = CANCELLED로 이력 보존 |

### 4-3. 전체 예매 플로우 (Happy Path)

```
1. 사용자 → 좌석 선택 요청
2. 서버 → Redis 분산락 획득 (Lettuce SETNX)
3. 서버 → Redis에 Hold 키 SET (TTL 5분, 값: userId)
        └ 동일 사용자의 중복 Hold 여부 확인 (어뷰징 방지)
4. 서버 → 분산락 해제 (UUID 검증 + Lua Script 원자적 삭제)
5. 서버 → Hold 성공 응답 (holdToken 발급)
6. 사용자 → 쿠폰 적용 (선택) + 결제 요청 (holdToken 포함)
7. 서버 → Mock PG 결제 요청
8. Mock PG → 결제 완료 웹훅 전송 (/api/mock-pg/webhook)
9. 서버 → 웹훅 수신
        └ Hold 유효성 검증 (Redis TTL 만료 여부)
        └ [TTL 만료] → 예매 실패 응답 (Race Condition 처리)
        └ [TTL 유효] → 쿠폰 사용 처리 (쿠폰락 획득 → 차감 → 해제)
                     → MySQL booking INSERT (CONFIRMED)
                     → Redis Hold 키 삭제
10. 서버 → 예매 확정 응답
```

### 4-4. 동시성 제어 락 획득 순서 (데드락 방지)

쿠폰 적용 + 좌석 확정이 동시에 일어날 때 **락 획득 순서를 반드시 아래 순서로 고정**한다.
순서가 다르면 Thread A ↔ Thread B 간 데드락이 발생할 수 있다.

```
① 좌석 Hold 락   (Redis: Lettuce SETNX)
② 쿠폰 사용 락   (Redis: Lettuce SETNX)
③ DB 트랜잭션    (MySQL: InnoDB)
```

> **원칙:** 항상 더 작은 범위(좌석)에서 더 큰 범위(쿠폰)로 락 획득. 해제는 역순.

> **⚠️ 도전 기능 전환 시:** Lettuce 기반 구현이 완성되고 테스트가 통과한 이후, Redisson으로 리팩토링하는 것을 도전 기능으로 진행한다. 순서를 반드시 지킬 것.

### 4-5. 어뷰징 방지 정책

| 정책 | 구현 방식 |
|------|-----------|
| 사용자 1인 최대 Hold 수 제한 | Redis `user-hold-count:{userId}` 카운터 INCR/DECR (TTL 300초), 4 이상이면 즉시 거부 (M-02 대응) |
| 동일 좌석 중복 Hold 방지 | Redis 분산락 내 `holdToken` 존재 여부 확인 |
| Hold 만료 후 결제 완료 수신 | 웹훅 수신 시 Redis TTL 검증 후 예매 실패 처리 |

## 6. 캐싱 전략 — 3단계 진화 흐름

> 발제 요구사항에 따라 캐싱은 아래 순서로 단계적으로 구현한다.
> **단계를 건너뛰지 말 것.** 각 단계의 한계를 직접 경험한 뒤 다음 단계로 이동하는 것이 학습 목적이다.

```
[1단계] v1 API — 캐시 없음
  └ 매 요청마다 MySQL 직접 조회
  └ 학습 목표: 캐시가 없을 때의 DB 부하와 응답 속도 측정

        ↓ 필수 구현

[2단계] v2 API — Caffeine 로컬 캐시 적용
  └ @Cacheable + Caffeine, TTL·maximumSize 설정
  └ 학습 목표: 캐시 적용 전후 성능 비교, 로컬 캐시의 한계 인지
  └ 한계: 서버가 2대 이상(Scale-out)이면 서버 간 캐시 불일치 발생
  └ **전환 기준:** v1 대비 응답 시간 50% 이상 단축 확인 후 3단계 진행

        ↓ 도전 구현

[3단계] v2 API 업그레이드 — Redis 원격 캐시로 교체
  └ RedisTemplate + Cache-Aside 패턴
  └ 학습 목표: 분산 환경에서 캐시 공유, Eviction 전략 설계
  └ v1 / v2(Caffeine) / v2(Redis) 3단계 성능 비교 보고서 작성
  └ **전환 방식:** `@ConditionalOnProperty(name = "cache.provider", havingValue = "redis")`
    application.yml의 `cache.provider` 값을 `caffeine` → `redis`로 변경하는 것만으로 전환
    (코드 변경 없이 설정만으로 전환 가능하도록 설계)
```

### 캐시 적용 대상 및 TTL 설계

| 캐시 대상 | 저장소 | TTL | 이유 |
|-----------|--------|-----|------|
| 이벤트 검색 결과 | Caffeine → Redis | 5분 | 자주 변경되지 않으나 조회 빈도 높음 |
| 인기 검색어 Top 10 | Redis ZSet | 실시간 (TTL 없음) | ZSet 점수로 실시간 관리, 별도 TTL 불필요 |
| 이벤트 상세 조회 | Caffeine | 10분 | 공연 정보는 변경 빈도 낮음 |

## 6-1. 데이터 볼륨 및 캐시 무효화 전략

> 검색 성능 비교(v1 vs v2 Caffeine vs v2 Redis)와 Redis ZSet 기반 인기 검색어 처리를 **의미 있게** 하기 위해 데이터 볼륨 확보와 캐시 무효화 설계에 집중한다.

### 1. 데이터 볼륨 전략
- **Venue**: ERD상 VENUE 테이블 유지 → 시드 데이터 **1개만** 생성 (`venue_id = 1`)
- **Event**: 최소 **5,000개 이상** 생성 (검색 성능 비교를 위한 충분한 데이터량)
- **Section**: 이벤트당 **3~6개** 생성
- **Seat**: 구역당 **100~150석** 규모로 생성  
  → 한 이벤트당 총 좌석 수: 약 **300~900석** (현실적인 공연장 규모)
- **목적**: 
  - v1(캐시 없음)에서 DB 부하를 실제로 체감
  - v2(Caffeine)와 v2(Redis) 성능 차이를 명확히 측정
  - 인기 검색어 ZSet이 실시간으로 의미 있는 랭킹을 형성할 수 있도록 함

### 2. 캐시 무효화 전략 (Cache Invalidation)
| 캐시 대상          | 무효화 트리거          | 구현 방식                     | 비고                  |
|--------------------|------------------------|-------------------------------|-----------------------|
| 이벤트 검색 결과   | 이벤트 등록·수정·삭제 | `@CacheEvict` 또는 Redis `DEL` | QueryDSL 동적 쿼리 키 생성 |
| 이벤트 상세 조회   | 이벤트 수정           | `@CacheEvict`                 | TTL 10분              |
| 인기 검색어 Top 10 | 검색 발생 시           | ZSet `ZINCRBY` (자동)         | TTL 없음              |

> **주의**: 캐시 무효화 로직을 소홀히 하면 v2(Redis) 단계에서 데이터 정합성 이슈가 발생할 수 있으므로,  
> 이벤트 서비스에 `@CacheEvict` AOP 또는 수동 무효화 메서드를 반드시 구현한다.

### 3. 시드 데이터 생성 계획
- `data.sql` 또는 `CommandLineRunner`를 통해 대량 데이터 자동 생성
- 이벤트 5,000개 + 각 이벤트별 섹션·좌석까지 한 번에 생성
- 테스트 시에도 동일한 대량 데이터를 사용해 성능 비교의 일관성 확보

## 7. 동시성 제어 전략 — 구현 단계 및 주의사항

> 발제 요구사항에 따라 동시성 제어도 아래 순서로 단계적으로 구현한다.

### 7-1. 구현 단계

```
[1단계] 동시성 테스트 코드 작성 (실패 확인)
  └ ExecutorService + CyclicBarrier로 100 Thread 동시 요청 시나리오
  └ 이 시점에 테스트는 반드시 실패해야 정상

        ↓

[2단계] Lettuce SETNX 기반 분산락 구현 (필수)
  └ DistributedLockProvider 인터페이스 정의 (Feature Flag 추상화)
  └ LettuceDistributedLock 구현체 + LockConfig (lock.provider=lettuce) 등록
  └ UUID 기반 본인 락만 해제 + Lua Script 원자적 삭제
  └ 락 획득 실패 전략 결정: Fail Fast (티켓팅 특성상 즉시 실패가 UX에 유리)
  └ 테스트 재실행 → 통과 확인

        ↓ 도전 구현

[3단계] Redisson 구현 + Feature Flag 전환
  └ RedissonDistributedLock 구현체 추가 + LockConfig에 redisson Bean 등록
  └ @RedisLock 커스텀 애노테이션 + @Aspect 구조로 락 코드 분리 (선택)
  └ 동일 테스트로 Redisson 검증 → application.yml lock.provider: lettuce → redisson 변경만으로 전환
```

### 7-2. 동시성 제어 대상 시나리오

TicketFlow는 **티켓 예매(좌석 Hold)**와 **쿠폰 발급** 두 가지를 모두 동시성 제어 대상으로 설계한다.

> **⚠️ 복잡도 주의사항**
>
> 두 가지 락을 동시에 관리하면 단일 락보다 구현 난이도가 높다.
> 특히 쿠폰 적용이 예매 플로우 내에서 이뤄질 경우 **두 개의 락이 중첩**되므로,
> 반드시 아래 순서를 지켜 데드락을 방지한다.
>
> **권장 구현 순서:**
> 1. 쿠폰 발급 단독 동시성 제어 먼저 구현 + 테스트 통과
> 2. 좌석 Hold 동시성 제어 구현 + 테스트 통과
> 3. 두 락의 중첩 시나리오(예매 시 쿠폰 적용) 통합 테스트
>
> 시간이 부족할 경우, 두 시나리오 중 **쿠폰 발급을 우선** 완성하고 좌석 Hold는 단순화하여 진행한다.

| 시나리오 | 락 키 | 실패 전략 |
|----------|-------|-----------|
| 좌석 임시 점유 | `lock:seat:{eventId}:{seatId}` | Fail Fast (즉시 "선점됨" 반환) |
| 쿠폰 발급 | `lock:coupon:{couponId}` | Retry with backoff (최대 3회) |
| 쿠폰 사용 (예매 내) | `lock:coupon-use:{couponId}:{userId}` | Fail Fast |

### 7-3. 락 비교 분석 (발표 및 README 필수 정리)

| 구분 | 낙관적 락 | 비관적 락 | 분산 락 (Lettuce) |
|------|----------|----------|------------------|
| 관리 주체 | JPA @Version | DB (InnoDB Row Lock) | Redis |
| 보호 범위 | DB 수정 시점 | DB 트랜잭션 범위 | 비즈니스 로직 전체 |
| 서버 Scale-out | 불가 | 불가 | 가능 |
| 성능 | 충돌 적을 때 유리 | 충돌 잦을 때 유리 | 외부 네트워크 비용 |
| 적용 시나리오 | 충돌 드문 수정 | 충돌 잦은 단일 서버 | 티켓팅·선착순 이벤트 |
