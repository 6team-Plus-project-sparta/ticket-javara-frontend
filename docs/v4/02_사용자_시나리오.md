## 페르소나 설계 원칙

TicketFlow의 페르소나는 단순한 "가상 사용자"가 아니라, **우리가 구현할 기능이 왜 필요한지를 증명하는 존재**다.
각 페르소나가 겪는 Pain Point가 곧 우리의 기술적 도전 과제와 1:1로 매핑된다.

| 페르소나 | 핵심 연결 기능 |
|----------|----------------|
| 페르소나 1 — 박서준 | 동시성 제어 (좌석 Hold + 분산락), 어뷰징 방지 |
| 페르소나 2 — 김하늘 | 검색 캐싱 (Caffeine → Redis), 인기 검색어, DB 인덱스 최적화 |
| 페르소나 3 — 이도현 | 선착순 쿠폰 발급 동시성, CS 채팅 (WebSocket), 배포 안정성 |
| 페르소나 4 — 최운영 (관리자) | 이벤트·쿠폰 등록, CS 채팅 응대, 예매 수동 확정 |

## 페르소나 1 — "열어라 티켓" 박서준

### 프로필

| 항목 | 내용 |
|------|------|
| 이름 | 박서준 |
| 나이 | 27세 |
| 직업 | 대기업 마케팅팀 사원 |
| 특징 | 아이돌 콘서트 현장 관람 마니아. 연간 8~10회 공연 관람. 티켓 오픈 시간에 맞춰 노트북 + 스마트폰 동시 접속하는 "티켓팅 전사". |

### Goal

> **원하는 좌석을 공정하게, 확실하게 선착순으로 확보하고 싶다.**

- 오픈 시간에 빠르게 좌석을 선택하면 그 좌석이 내 것으로 보장되길 원함
- 결제 도중 다른 사람에게 좌석을 뺏기는 경험을 하고 싶지 않음
- 한 번 잡은 좌석은 결제가 끝날 때까지 안전하게 유지되길 기대

### Pain Point

| # | 기존 서비스에서 겪는 불편함 | 연결 기능 |
|---|---------------------------|-----------|
| 1 | 좌석 선택 후 결제창까지 갔는데 "이미 예매된 좌석입니다" 오류 발생 → 결제 취소 + 재선택 반복 | **좌석 Hold (임시 점유)** — 선택 시점에 5분 TTL 점유로 결제 중 좌석 유실 방지 |
| 2 | 매크로·다중 계정 사용자가 좌석을 싹쓸이해서 일반 사용자는 기회가 없음 | **어뷰징 방지 정책** — 1인 최대 4석 Hold 제한, 동일 좌석 중복 Hold 차단 |
| 3 | 서버 과부하로 오픈 직후 사이트 자체가 다운되어 접속조차 불가 | **동시성 제어 (분산락)** — Redis 분산락으로 직렬 처리, DB 과부하 방지 |

### 주요 사용 시나리오

#### 시나리오 1-1: 콘서트 티켓 오픈 직후 좌석 선점

```
상황: 금요일 오후 8시, 인기 아이돌 콘서트 티켓 오픈.
      서준이는 오픈 10분 전부터 이벤트 상세 페이지에서 대기 중.

1. 오후 8시 00분 00초 — 좌석 선택 화면 진입
2. 원하는 좌석(A구역 3열 15번) 클릭
3. 서버는 Redis 분산락 획득 → Hold 키 SET (TTL 5분) → 락 해제
4. "좌석이 임시 점유되었습니다. 5분 내 결제를 완료해주세요." 안내
5. 결제 정보 입력 → Mock PG 결제 요청
6. 웹훅 수신 → Hold 유효성 검증 → BOOKING CONFIRMED + ACTIVE_BOOKING INSERT
7. "예매가 완료되었습니다!" 확인 화면

→ 기술 연결: 분산락(Lettuce SETNX), 좌석 Hold TTL, ACTIVE_BOOKING 중복 방어선
```

#### 시나리오 1-2: 동시 접속자와의 좌석 경쟁

```
상황: 서준이와 다른 사용자 999명이 동시에 같은 좌석을 클릭.

1. 서준이 포함 1,000명이 동시에 A구역 3열 15번 선택 요청
2. Redis 분산락에 의해 단 1명만 Hold 획득 성공
3. [서준이가 성공한 경우] → Hold 성공, 결제 진행
4. [서준이가 실패한 경우] → 즉시 "이미 선점된 좌석입니다" 응답 (Fail Fast)
   → 다른 좌석을 바로 선택할 수 있도록 안내

→ 기술 연결: Fail Fast 전략, 분산락 직렬 처리, 즉시 응답 UX
```

#### 시나리오 1-3: Hold 만료 후 좌석 자동 해제

```
상황: 서준이가 좌석을 점유했지만, 결제 수단 문제로 5분을 초과.

1. 좌석 Hold 성공 (TTL 5분 시작)
2. 카드 한도 초과로 다른 카드 찾는 사이 5분 경과
3. Redis TTL 만료 → Hold 키 자동 삭제 → 좌석 AVAILABLE로 복귀
4. 뒤늦게 결제 시도 → "점유 시간이 만료되었습니다. 좌석을 다시 선택해주세요."
5. 해당 좌석은 다른 대기 사용자가 선점 가능

→ 기술 연결: Redis TTL 자동 만료, 웹훅 수신 시 Hold 유효성 검증
```

#### 시나리오 1-4: 친구들과 나란히 앉기 위한 멀티 좌석 선택 ⭐

```
상황: 서준이가 친구 3명과 함께 콘서트를 관람하기 위해 4석을 한 번에 예매.

1. 좌석 선택 화면에서 A구역 3열 15, 16, 17, 18번 순서대로 클릭
2. 각 좌석 클릭 시마다 서버는 개별적으로 분산락 획득 → Hold 키 SET → 락 해제
   - hold:{eventId}:15 = userId (TTL 5분)
   - hold:{eventId}:16 = userId (TTL 5분)
   - hold:{eventId}:17 = userId (TTL 5분)
   - hold:{eventId}:18 = userId (TTL 5분)
   - user-hold-count:{userId} = 4 (TTL 300초 카운터, INCR 4회)
3. 4석 모두 Hold 성공 → holdToken 4개 보유
4. "4석이 임시 점유되었습니다. 5분 내 결제를 완료해주세요." 안내
5. 쿠폰 없이 결제 요청 (holdTokens: [token15, token16, token17, token18])
6. 서버: 4개 holdToken 일괄 검증 → ORDER 생성 (total_amount = 4 × 110,000 = 440,000원)
7. 각 좌석별 BOOKING 4건 생성 (PENDING)
8. Mock PG 결제 → 웹훅 수신 → DB 트랜잭션:
   - ORDER → CONFIRMED
   - BOOKING 4건 → CONFIRMED
   - ACTIVE_BOOKING 4건 INSERT (seat_id 15, 16, 17, 18)
9. 모든 Hold 키 삭제, user-hold-count:{userId} DECR (0 이하 시 DEL)
10. "4석 예매가 완료되었습니다!" 확인 화면

→ 기술 연결: 다건 Hold 관리 (user-hold-count 카운터 방식, M-02), ORDER ↔ BOOKING 1:N 구조,
             ACTIVE_BOOKING 다건 INSERT, 멱등성 보장 (웹훅 중복 처리 방지)

[어뷰징 방지 검증 포인트]
- 5번째 좌석 클릭 시 user-hold-count:{userId} ≥ 4 → 즉시 400 "최대 4석까지만 선택할 수 있습니다."
- Lua Script 원자적 체크로 동시 요청 시에도 5석 초과 방지
```

## 페르소나 2 — "탐색하는 문화인" 김하늘

### 프로필

| 항목 | 내용 |
|------|------|
| 이름 | 김하늘 |
| 나이 | 34세 |
| 직업 | IT기업 UX 디자이너 |
| 특징 | 뮤지컬·연극을 주로 관람. 특정 공연을 정해두고 오기보다, 장르·날짜·가격 조건으로 "뭐 볼 만한 거 없나" 탐색하는 스타일. 주 1회 이상 앱에서 공연을 검색. |

### Goal

> **내 취향에 맞는 공연을 빠르고 편하게 찾아서 비교하고 싶다.**

- 장르, 날짜, 가격대로 필터링해서 한눈에 비교
- 검색 결과가 즉각적으로 나와야 함 (로딩에 3초 이상 걸리면 이탈)
- 요즘 사람들이 뭘 많이 보는지 트렌드 파악

### Pain Point

| # | 기존 서비스에서 겪는 불편함 | 연결 기능 |
|---|---------------------------|-----------|
| 1 | 검색할 때마다 3~5초 로딩. 조건 바꿔서 재검색하면 또 기다림 → 탐색 피로 | **Caffeine 로컬 캐싱** — 동일 검색어 재요청 시 DB 조회 없이 즉시 응답 |
| 2 | "요즘 인기 공연"을 알 수 있는 방법이 없음. 매번 직접 찾아야 함 | **인기 검색어 (Redis ZSet)** — 실시간 Top 10 검색어로 트렌드 파악 |
| 3 | 공연이 수천 개인데 정렬·필터가 느림. 페이지 넘길 때마다 답답 | **DB 인덱스 최적화 (EXPLAIN)** — 복합 인덱스로 LIKE + 정렬 쿼리 최적화 |

### 주요 사용 시나리오

#### 시나리오 2-1: 장르별 공연 탐색 및 캐시 히트

```
상황: 하늘이는 이번 주말 볼 뮤지컬을 찾고 있다.

1. 검색창에 "뮤지컬" 입력 후 검색
2. [첫 번째 요청] DB 조회 → 결과 반환 + Caffeine 캐시 저장 (TTL 5분)
3. 결과를 훑어보다가 "뮤지컬" 조건 그대로 2페이지 클릭
4. [두 번째 요청] Caffeine 캐시 히트 → DB 조회 없이 즉시 응답 (< 50ms)
5. 응답 속도 차이를 체감하며 계속 탐색

→ 기술 연결: 검색 v2 API (Caffeine @Cacheable), 캐시 키 설계 (검색어+페이지+정렬)
```

#### 시나리오 2-2: 인기 검색어로 트렌드 파악

```
상황: 하늘이는 특별히 보고 싶은 게 없어서 "요즘 뭐가 핫하지?" 궁금.

1. 앱 메인 화면 진입 → 인기 검색어 Top 10 노출
2. "오페라의 유령", "세븐틴 콘서트", "KBO 올스타전" 등 실시간 랭킹 확인
3. "오페라의 유령" 클릭 → 해당 검색어로 자동 검색
4. 검색 시 Redis ZSet 점수 +1 (ZINCRBY) → 인기 검색어 순위 실시간 반영

→ 기술 연결: Redis ZSet (ZINCRBY + ZREVRANGE), 인기 검색어 API, 실시간 순위 갱신
```

#### 시나리오 2-3: 복잡한 조건 검색과 DB 성능 체감

```
상황: 하늘이는 "서울, 이번 달, 5만원 이하 뮤지컬"로 검색.

1. 다중 조건 검색: 장르=뮤지컬, 지역=서울, 날짜=이번 달, 가격≤50,000원
2. [인덱스 미적용 시] Full Table Scan → 응답 800ms
3. [EXPLAIN 분석 후 복합 인덱스 적용] → 응답 120ms
4. 결과 목록에서 날짜순 정렬, 가격순 정렬 전환도 빠르게 처리
5. 하늘이: "오, 여기는 검색이 빠르네" (이탈하지 않음)

→ 기술 연결: EXPLAIN 기반 쿼리 분석, 복합 인덱스 (category, event_date),
           v1 vs v2 성능 비교 리포트
```

## 페르소나 3 — "알뜰한 첫 직장인" 이도현

### 프로필

| 항목 | 내용 |
|------|------|
| 이름 | 이도현 |
| 나이 | 24세 |
| 직업 | 중소기업 경영지원팀 신입사원 |
| 특징 | 야구 관람이 유일한 취미. 월급이 많지 않아서 할인 쿠폰이나 프로모션에 매우 민감. 처음 사용하는 앱에서 문제가 생기면 바로 CS 문의를 넣는 타입. |

### Goal

> **할인 쿠폰을 놓치지 않고 챙겨서, 가장 저렴하게 야구 티켓을 사고 싶다.**

- 선착순 쿠폰 오픈 시간에 맞춰 빠르게 발급받기
- 쿠폰 적용해서 실제 결제 금액 절약
- 문제가 생기면 빠르게 상담받아 해결

### Pain Point

| # | 기존 서비스에서 겪는 불편함 | 연결 기능 |
|---|---------------------------|-----------|
| 1 | 선착순 쿠폰 발급 버튼 눌렀는데 "이미 소진되었습니다" → 근데 다른 사람은 받았다고 함. 수량 관리가 정확하지 않음 | **쿠폰 발급 동시성 제어** — 분산락으로 수량 초과 발급 0건 보장 |
| 2 | 예매 중 쿠폰 적용했는데 최종 결제에서 "쿠폰이 이미 사용되었습니다" 오류 → 쿠폰 날림 | **쿠폰 사용 락 + 예매 플로우 내 락 순서** — 쿠폰-좌석 데드락 방지, 원자적 쿠폰 사용 처리 |
| 3 | CS 문의를 이메일로 보냈는데 3일째 답변 없음. 전화는 항상 통화중 | **WebSocket(STOMP) CS 채팅** — 실시간 양방향 채팅으로 즉시 문의·응답 |

### 주요 사용 시나리오

#### 시나리오 3-1: 선착순 할인 쿠폰 발급 경쟁

```
상황: 정오 12시, TicketFlow 신규 가입 기념 5,000원 할인 쿠폰 100장 한정 오픈.

1. 도현이 포함 500명이 12시 00분 00초에 동시 발급 요청
2. [중복 체크 먼저] UserCoupon 테이블에서 (userId, couponId) 존재 확인 (락 획득 전)
   → 이미 받은 사람은 즉시 409, 락 비용 없이 차단
3. Redis Atomic DECR으로 수량 차감 (남은 수량 0이면 즉시 차단)
4. [처음 받는 사람, 재고 있음] UserCoupon INSERT (status=ISSUED)
5. 도현이: "쿠폰이 발급되었습니다!"
6. [101번째부터] "쿠폰이 모두 소진되었습니다." (정확히 100장만 발급)

→ 기술 연결: 중복 체크 선행 (락 바깥), Redis Atomic DECR,
           동시성 테스트 (100 Thread)로 재고 초과 0건 검증
```

#### 시나리오 3-2: 쿠폰 적용 + 티켓 예매 통합 플로우

```
상황: 도현이가 KBO 야구 경기 티켓을 5,000원 쿠폰 적용하여 예매.

1. 외야 자유석 좌석 선택 → 좌석 Hold 성공 (Hold TTL 5분 시작)
2. 보유 쿠폰 목록에서 5,000원 할인 쿠폰 선택 → 적용
3. 결제 요청 (holdToken + couponId 포함)
4. 서버: holdToken 검증 → 쿠폰 유효성 검증
5. ORDER 생성 (total_amount=35,000, discount_amount=5,000, final_amount=30,000)
6. BOOKING 생성 (PENDING, original_price=35,000)
7. Mock PG 결제 요청 (30,000원)
8. 웹훅 수신 → Hold 재검증 → 쿠폰 사용 락 획득 + 재검증 → DB 트랜잭션:
   - ORDER, BOOKING → CONFIRMED
   - ACTIVE_BOOKING INSERT
   - USER_COUPON → USED
9. Hold 키 삭제 → 쿠폰 락 해제 (finally)
10. "35,000원 → 30,000원 결제 완료!"

→ 기술 연결: 락 획득 순서 (① Hold 검증 → ② 쿠폰 락 → ③ DB), 데드락 방지,
           ACTIVE_BOOKING으로 좌석 확정 보장
```

#### 시나리오 3-3: 예매 문제 발생 후 실시간 CS 채팅

```
상황: 도현이가 결제는 됐는데 예매 내역에 티켓이 안 보인다.

1. 앱 하단 "문의하기" 버튼 클릭 → CS 채팅방 입장
2. WebSocket(STOMP) 연결 → 관리자 채팅방에 실시간 알림
3. 도현: "결제는 됐는데 예매 내역에 티켓이 없어요. 주문번호 BK-20260409-001입니다."
4. 관리자: (3분 내 응답) "확인 중입니다. 결제 확인되어 수동 확정 처리하겠습니다."
5. 관리자가 관리자 대시보드에서 해당 bookingId를 선택 →
   PATCH /api/admin/bookings/{bookingId}/confirm 호출
   → ORDER, BOOKING → CONFIRMED, ACTIVE_BOOKING INSERT 처리
6. 처리 완료 후 도현이에게 채팅으로 완료 알림 전송
7. 도현: "감사합니다!" → 채팅방 종료

→ 기술 연결: WebSocket + STOMP 프로토콜, 사용자-관리자 1:1 채팅방,
           실시간 메시지 송수신, 수동 확정 API (ADMIN 권한)
```

## 페르소나 4 — "꼼꼼한 운영자" 최운영 (관리자)

### 프로필

| 항목 | 내용 |
|------|------|
| 이름 | 최운영 |
| 나이 | 32세 |
| 직업 | TicketFlow 운영팀 CS 담당자 |
| 특징 | 공연 이벤트 등록, 쿠폰 캠페인 세팅, 고객 CS 문의 처리를 하루에 수십 건 담당. 시스템 이슈 발생 시 빠르게 수동 조치하는 역할. ADMIN 권한 보유. |

### Goal

> **시스템 이슈를 최소화하고, 문제 발생 시 빠르게 수동 조치해서 고객 피해를 막고 싶다.**

- 이벤트·쿠폰을 정확하게 등록해서 오픈 당일 혼선 없이 운영
- CS 채팅을 통해 고객 문의를 실시간으로 확인하고 빠르게 응대
- PG 웹훅 실패 등 예외 상황 발생 시 수동으로 예매를 확정 처리

### Pain Point

| # | 기존 운영에서 겪는 불편함 | 연결 기능 |
|---|---------------------------|-----------| 
| 1 | 고객이 결제는 완료됐는데 예매 상태가 PENDING에 멈춰 있는 경우 수동으로 DB에 직접 접근해야 함 | **수동 확정 API** — 관리자가 대시보드에서 bookingId를 확정 처리 |
| 2 | 고객 문의가 이메일로만 들어와서 실시간 응대 불가 | **CS 채팅 (WebSocket STOMP)** — 실시간 문의 수신 및 응답 |
| 3 | 쿠폰 캠페인 등록 후 수량이 잘못 세팅되어도 확인 수단이 없음 | **관리자 쿠폰 등록 API** — 총 수량, 유효기간, 할인액 명시적 설정 |

### 주요 사용 시나리오

#### 시나리오 4-1: 공연 이벤트 등록

```
상황: 새 콘서트 "아이유 2026 TOUR" 티켓 판매를 위해 이벤트를 등록.

1. 관리자 대시보드 → 이벤트 등록 화면 접근 (ADMIN 권한 필요)
2. 이벤트 정보 입력: 제목, 카테고리(CONCERT), 날짜, 회차(round_number=1), 예매 오픈/마감 시각
3. 구역(Section) 설정: VIP석(150,000원, 100석), R석(110,000원, 200석), S석(80,000원, 300석)
4. POST /api/admin/events 호출 → EVENT + SECTION + SEAT 일괄 생성
5. "이벤트가 등록되었습니다." 확인

→ 기술 연결: 이벤트·구역·좌석 일괄 생성 (venue_id=1 고정), ADMIN 권한 검증
```

#### 시나리오 4-2: 쿠폰 캠페인 등록

```
상황: 오픈 기념 5,000원 할인 쿠폰 100장을 정오 12시부터 발급 시작.

1. 관리자 대시보드 → 쿠폰 등록 화면 접근
2. 쿠폰 정보 입력: 이름("오픈 기념 5,000원 할인"), 할인액(5,000원), 총 수량(100),
   발급 시작(2026-04-10T12:00:00), 만료(2026-05-31T23:59:59)
3. POST /api/admin/coupons 호출 → COUPON 레코드 생성 + Redis remaining_quantity 초기화
4. 정오 이후 사용자들이 선착순 발급 가능 상태 확인

→ 기술 연결: 쿠폰 등록 API (ADMIN 권한), Redis remaining_quantity 초기화
```

#### 시나리오 4-3: CS 채팅 응대 및 수동 예매 확정

```
상황: 고객(이도현)의 CS 채팅 문의가 들어옴. 결제 완료됐으나 BOOKING이 PENDING 상태.

1. 관리자 대시보드 좌측 채팅방 목록에 신규 문의 알림 수신 (/sub/chat/rooms)
2. GET /api/admin/chat/rooms?status=OPEN 으로 현재 OPEN 채팅방 목록 조회
3. 이도현의 채팅방 클릭 → 채팅 이력 확인
4. "확인 중입니다. 수동 확정 처리하겠습니다." 답장 전송
5. 해당 bookingId 확인 → PATCH /api/admin/bookings/{bookingId}/confirm 호출
   → 단일 트랜잭션: ORDER/BOOKING → CONFIRMED, ACTIVE_BOOKING INSERT
6. 확정 완료 후 채팅으로 결과 알림 전송 → 채팅방 종료

→ 기술 연결: 관리자 채팅방 목록 조회 REST API, WebSocket STOMP 실시간 수신,
           수동 확정 API (SELECT ... FOR UPDATE 비관적 락으로 취소 충돌 방지)
```
