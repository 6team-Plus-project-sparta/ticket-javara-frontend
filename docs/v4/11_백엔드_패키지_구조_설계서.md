# 11. 백엔드 패키지 구조 설계서

> **버전**: v1.1
> **패키지 루트**: `com.example.ticketjavara`
> **설계 원칙**: 도메인형(Domain-Driven) + 계층형(Layered) 혼합 구조
>
> **핵심 설계 결정 요약**
>
> | 항목 | 결정 | 이유 |
> |------|------|------|
> | 구조 방식 | 도메인형 최상위 + 계층형 내부 | 도메인 간 결합도 최소화, 팀원별 도메인 독립 개발 |
> | 분산락 전용 Facade 분리 | `HoldLockFacade` | `@Transactional` ↔ Lettuce Lock 생명주기 충돌 방지 |
> | 분산락 인터페이스 추상화 | `DistributedLockProvider` | Feature Flag(`lock.provider`)로 Lettuce↔Redisson 무중단 전환 (ADR-002) |
> | 락 전환 설정 | `LockConfig` (`@ConditionalOnProperty`) | 캐시(`CacheConfig`)와 동일 패턴 — yml 한 줄로 전환·롤백 |
> | QueryDSL Custom 분리 | `EventRepositoryCustom` | 동적 검색 쿼리를 JPA Repository와 분리 |
> | 락 공통 모듈 분리 | `global/lock` | 팀원 C(좌석)·D(쿠폰)가 동일 분산락 로직 공유 |
> | WebSocket Config 위치 | `global/config` | 인프라 설정은 도메인이 아닌 전역 관심사로 분류 |

## 1. 전체 패키지 트리

```
src/main/java/com/example/ticketflow/
│
├── global/                         # 앱 전체 공통 관심사
│   ├── config/                     # 인프라 · 프레임워크 설정
│   ├── security/                   # JWT 인증 필터 체인
│   ├── lock/                       # Lettuce SETNX 분산락 공통 모듈
│   ├── exception/                  # 공통 예외 정의 · 핸들러
│   ├── common/                     # 공통 응답 DTO · BaseEntity
│   └── util/                       # 범용 유틸리티
│
└── domain/                         # 비즈니스 도메인
    ├── auth/                       # 인증 (회원가입 · 로그인 · JWT)  ← 팀원 A
    ├── user/                       # 회원 프로필 · 내 정보          ← 팀원 A
    ├── event/                      # 이벤트 CRUD · 구역 · 좌석      ← 팀원 B
    ├── search/                     # 이벤트 검색 v1/v2 · 인기검색어  ← 팀원 B
    ├── booking/                    # 좌석 Hold · 주문 · 예매 확정   ← 팀원 C
    ├── coupon/                     # 쿠폰 발급 · 적용 · 조회        ← 팀원 D
    └── chat/                       # CS 채팅 (WebSocket/STOMP)     ← 팀원 D
```

```
domain/event/

│   ├── EventController.java        # GET  /api/events           (목록, 💾 캐시 적용 대상 — @Cacheable은 EventRepository에 위치)

├── dto/

│   │   │                               #   saleEndAt, roundNumber, sections[] }
│   │   └── SectionCreateDto.java       # { sectionName, price, rowCount, colCount }

│       ├── EventSummaryResponseDto.java  # 목록용 { eventId, title, category, eventDate,

│   │                               # 💾 @Cacheable("event-detail") TTL 10분 — 이벤트 상세 조회 캐싱

### 3-4. `search` — 검색 · 인기검색어 (팀원 B)

캐싱 3단계 진화(v1 → Caffeine → Redis)와 Redis ZSet 인기검색어가 집중된 도메인.

```
domain/search/
├── controller/
│   └── SearchController.java       # GET  /api/v1/events/search  (v1: 캐시 없음, 기준선)
│                                   # GET  /api/v2/events/search  (v2: Caffeine → Redis)
│                                   # GET  /api/search/popular    (인기 검색어 Top 10)
│                                   # POST /api/search/popular/click (클릭 점수 반영 🔐)
├── dto/
│   ├── request/
│   │   └── SearchRequestDto.java   # { keyword, category, startDate, endDate,
│   │                               #   minPrice, maxPrice, page, size, sort }
│   └── response/
│       ├── SearchResponseDto.java  # Page<EventSummary> + X-Cache: HIT/MISS 헤더
│       └── PopularKeywordDto.java  # { rank, keyword, count }
├── repository/
│   ├── EventSearchRepositoryCustom.java  # [Interface] QueryDSL 동적 쿼리 계약
│   └── EventSearchRepositoryImpl.java    # [Class] BooleanBuilder 동적 검색 구현
│                                         #   가격 필터: EXISTS 서브쿼리 (ADR-009)
│                                         #   "구역 중 하나라도 해당 범위면 이벤트 노출"
└── service/
    └── SearchService.java          # v1: MySQL 직접 조회 (캐시 없음, 응답 시간 로그 필수)
                                    # v2: @Cacheable("event-search") TTL 5분
                                    #   캐시 키: {keyword}:{category}:{startDate}:...
                                    # 인기 검색어: Redis ZINCRBY + ZREVRANGE Top 10
                                    # 클릭 이벤트: 1시간 1회 Redis 중복 방지
```

> **캐싱 3단계 전환 방식 (ADR-006)**:
> `application.yml`의 `cache.provider: caffeine` → `cache.provider: redis` 한 줄 변경만으로
> `@ConditionalOnProperty`가 CacheManager Bean을 자동 전환. 코드 수정 없음.
>
> **가격 필터 EXISTS 서브쿼리 (ADR-009)**:
> ```java
> // BooleanBuilder — 가격 필터 구현
> if (minPrice != null || maxPrice != null) {
>     builder.and(JPAExpressions.selectOne()
>         .from(section)
>         .where(section.eventId.eq(event.id)
>             .and(section.price.goe(minPrice))
>             .and(section.price.loe(maxPrice)))
>         .exists());
> }
> ```

### 3-5. `booking` — 좌석 Hold · 예매 · 동시성 제어 ⭐⭐ (팀원 C)

분산락(Lettuce SETNX), ACTIVE_BOOKING, 웹훅 처리가 집중된 핵심 도메인.

```
domain/booking/
├── controller/
│   ├── HoldController.java         # POST   /api/events/{eventId}/seats/{seatId}/hold  (분산락 ⚠️)
│   │                               # DELETE /api/events/{eventId}/seats/{seatId}/hold
│   │                               # GET    /api/events/{eventId}/seats               (좌석 상태 조회)
│   ├── OrderController.java        # POST /api/orders              (주문 생성 ⚠️)
│   │                               # GET  /api/orders/{orderId}    (주문 상세 조회 · 폴링)
│   │                               # POST /api/orders/{orderId}/cancel (주문 취소 ⚠️)
│   ├── WebhookController.java      # POST /api/mock-pg/webhook     (PG 웹훅 수신 🔓)
│   └── BookingAdminController.java # PATCH /api/admin/bookings/{bookingId}/confirm (수동 확정 👑)
│                                   # GET   /api/users/me/bookings                  (예매 내역)
│
├── facade/
│   └── HoldLockFacade.java         # ⭐ Lettuce 분산락 전용 Facade (@Component, 트랜잭션 없음)
│                                   #   tryLock(lockKey, uuid, ttl=3s) → 실패 시 즉시 409 Fail Fast
│                                   #   holdService.processHold() 호출 (내부에서 @Transactional)
│                                   #   finally: unlockLua(lockKey, uuid)  ← COMMIT 이후 해제 보장
│
├── dto/
│   ├── request/
│   │   ├── OrderCreateRequestDto.java  # { holdTokens: List<String>, couponId?: Long }
│   │   └── WebhookRequestDto.java      # { orderId, paymentStatus, paymentKey, paidAmount }
│   └── response/
│       ├── HoldResponseDto.java        # { holdToken, expiresAt }
│       ├── SeatStatusDto.java          # { seatId, rowName, colNum, status } AVAILABLE|ON_HOLD|CONFIRMED
│       ├── OrderResponseDto.java       # { orderId, status, finalAmount }
│       └── OrderDetailResponseDto.java # { orderId, status, bookings[], payment, couponUsed }
│
├── entity/
│   ├── Order.java                  # ORDER 테이블
│   │                               #   user_id FK, user_coupon_id FK(Nullable)
│   │                               #   total_amount, discount_amount, final_amount
│   │                               #   status: PENDING | CONFIRMED | CANCELLED | FAILED
│   ├── Booking.java                # BOOKING 테이블
│   │                               #   order_id FK, user_id FK(역정규화), seat_id FK
│   │                               #   event_id FK(역정규화: 다회차 구별용)
│   │                               #   original_price(결제 당시 원가 스냅샷)
│   │                               #   ticket_code UK(Nullable: "TF-연도-좌석정보-UUID앞4자리")
│   │                               #   status: PENDING | CONFIRMED | CANCELLED | FAILED
│   ├── ActiveBooking.java          # ACTIVE_BOOKING 테이블
│   │                               #   seat_id PK (중복 CONFIRMED DB 레벨 원천 차단)
│   │                               #   booking_id UK FK
│   └── Payment.java                # PAYMENT 테이블
│                                   #   order_id FK, payment_key UK, method, paid_amount
│                                   #   status: SUCCESS | FAILED | REFUNDED
│
├── enums/
│   ├── OrderStatus.java            # PENDING | CONFIRMED | CANCELLED | FAILED
│   ├── BookingStatus.java          # PENDING | CONFIRMED | CANCELLED | FAILED
│   └── PaymentStatus.java          # SUCCESS | FAILED | REFUNDED
│
├── repository/
│   ├── OrderRepository.java        # findByIdAndUserId() — 본인 주문 조회
│   ├── BookingRepository.java      # findByOrderId() — 주문 내 예약 목록
│   │                               # findByUserId(Pageable) — 내 예매 내역 (Offset 페이징)
│   ├── ActiveBookingRepository.java# existsBySeatId() — CONFIRMED 여부 확인
│   │                               # deleteBySeatId() — 취소 시 좌석 해제
│   └── PaymentRepository.java      # findByOrderId() — 결제 기록 조회
│
└── service/
    ├── HoldService.java            # Hold 획득 · 해제 핵심 로직 (@Transactional)
    │                               # 처리 순서:
    │                               #   1. user-hold-count:{userId} 확인 (4 이상 → 400)
    │                               #   2. ACTIVE_BOOKING 확인 (CONFIRMED → 409)
    │                               #   3. hold:{eventId}:{seatId} 확인 (ON_HOLD → 409)
    │                               #   4. Redis SET hold/{holdToken/user-hold-count 3키 저장
    ├── OrderService.java           # 주문 생성 (holdToken 역조회 + 검증 + Hold TTL +5분 연장)
    │                               # 주문 취소 (SELECT FOR UPDATE 4개 테이블)
    │                               # 락 획득 순서: ① 좌석 Hold → ② 쿠폰 사용 → ③ DB 트랜잭션
    └── WebhookService.java         # PG 웹훅 처리 (멱등성 보장)
                                    # Hold 재검증 → 쿠폰 락 → 단일 DB 트랜잭션:
                                    #   ORDER/BOOKING → CONFIRMED
                                    #   ticket_code 생성 및 저장
                                    #   ACTIVE_BOOKING INSERT (PK 중복 시 롤백)
                                    #   USER_COUPON → USED
                                    #   PAYMENT 레코드 생성
                                    # finally: hold/holdToken 키 삭제, 쿠폰 락 해제
```

> **HoldLockFacade 분리 이유 (면접 포인트)**:
>
> ```
> [잘못된 구조]                        [올바른 구조]
> HoldService {                        HoldLockFacade {         ← 락 담당 (트랜잭션 없음)
>   @Transactional                       lock.tryLock()
>   processHold() {                      holdService.processHold() ← @Transactional
>     lock.tryLock()                     lock.unlock()          ← COMMIT 이후 해제 보장
>     ... DB 처리 ...               }
>     lock.unlock()  ← 문제!         HoldService {             ← DB 처리 담당
>   }                                  @Transactional
> }                                    processHold() { ... }
>                                    }
>
> 문제: COMMIT 전에 unlock → 다른 스레드가 아직 반영 전 DB 상태를 읽음
> 해결: Facade가 트랜잭션 외부에서 lock/unlock 생명주기를 감싸도록 분리
> ```
>
> **좌석 상태 판단 로직 (ADR-003)**:
> ```
> 1. ACTIVE_BOOKING에 seat_id 존재 → CONFIRMED
> 2. Redis hold:{eventId}:{seatId} 키 존재 → ON_HOLD
> 3. 그 외 → AVAILABLE
> ※ SEAT 테이블에 status 컬럼 없음 (MySQL Partial Index 미지원으로 ACTIVE_BOOKING 분리)
> ```
>
> **Hold Redis 키 구조 (ADR-013)**:
> ```
> SET hold:{eventId}:{seatId}     = {userId}                      EX 300
> SET holdToken:{uuid}            = "{eventId}:{seatId}:{userId}" EX 300
> INCR user-hold-count:{userId}                                   EX 300
> ```

Redis DECR + Lua Script 원자적 수량 차감이 핵심인 도메인.

```
domain/coupon/
├── controller/
│   └── CouponController.java       # POST /api/admin/coupons           (쿠폰 등록 👑)
│                                   # POST /api/coupons/{couponId}/issue (발급 🔐 ⚠️ Redis Atomic)
│                                   # GET  /api/users/me/coupons         (내 쿠폰 목록)
│
├── dto/
│   ├── request/
│   │   └── CouponCreateRequestDto.java  # { name, discountAmount, totalQuantity,
│   │                                    #   startAt, expiredAt }
│   └── response/
│       └── UserCouponResponseDto.java   # { userCouponId, couponName, discountAmount, expiredAt }
│
├── entity/
│   ├── Coupon.java                 # COUPON 테이블
│   │                               #   remaining_quantity — Redis 유실 대비 Source of Truth
│   │                               #   start_at, expired_at
│   └── UserCoupon.java             # USER_COUPON 테이블
│                                   #   user_id FK, coupon_id FK
│                                   #   UNIQUE(user_id, coupon_id) — 중복 발급 방지
│                                   #   status: ISSUED | USED
│                                   #   @Version — 낙관적 락 (예매 플로우 내 쿠폰 사용 시)
│
├── enums/
│   └── UserCouponStatus.java       # ISSUED | USED
│
├── repository/
│   ├── CouponRepository.java       # findByIdWithLock() — DB fallback 시 SELECT FOR UPDATE
│   └── UserCouponRepository.java   # existsByUserIdAndCouponId() — 중복 발급 선행 체크 (락 밖)
│                                   # findByUserIdAndStatus() — 내 쿠폰 목록
│
└── service/
    └── CouponService.java          # 쿠폰 등록 → COUPON INSERT + Redis remaining 초기화
                                    # 발급 처리 순서 (ADR-008: 분산락 없음):
                                    #   1. DB 중복 체크 (락 바깥 — 빠른 실패, 부하 감소)
                                    #   2. Redis Lua Script 원자적 DECR (음수 방어)
                                    #      → 결과 < 0: INCR 원상복구 + 409 COUPON_EXHAUSTED
                                    #   3. UserCoupon INSERT (status=ISSUED)
                                    #   4. MySQL remaining_quantity 즉시 -1 동기화 (m-06)
                                    # Redis 장애 시 Fallback:
                                    #   SELECT * FROM coupon WHERE id=? FOR UPDATE
```

> **Redis DECR 단독 사용 이유 (ADR-008)**:
> DECR 명령 자체가 이미 원자적 연산. 추가 분산락은 불필요한 이중 잠금으로
> 초당 처리량 저하 (분산락이 직렬화 강제). Lua Script로 음수 방어 처리.
>
> **Lua Script — 음수 방어**:
> ```lua
> local val = redis.call('DECR', KEYS[1])
> if val < 0 then
>     redis.call('INCR', KEYS[1])  -- 원상복구
>     return -1                     -- 소진 신호
> end
> return val
> ```

                    └─▶ @Transactional COMMIT
              └─▶ HoldLockFacade        ← finally: Lua Script unlock()  (COMMIT 이후)
  └─▶ Response: 200 OK { holdToken, expiresAt }


[요청 처리 흐름 — PG 웹훅 수신]
