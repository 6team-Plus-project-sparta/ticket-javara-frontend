## 문서 구조 안내

이 문서는 유스케이스 명세서(UC-001 ~ UC-012)를 **개발자가 바로 구현할 수 있는 기능 단위**로 세분화한다.
각 기능(FN)은 하나의 API 엔드포인트 또는 독립된 비즈니스 로직에 대응된다.

### 기능 ID 체계

```
FN-[도메인 약어]-[순번]

도메인 약어:
  AUTH  = 회원/인증
  EVT   = 이벤트 관리
  SRCH  = 검색
  SEAT  = 좌석/Hold
  BK    = 예매(Order & Booking)
  CPN   = 쿠폰/프로모션
  CHAT  = CS 채팅
```

### 필수/도전 표기

| 표기 | 의미 |
|------|------|
| 🟢 필수 | Must Have — 3주 내 반드시 완성 |
| 🔵 도전 | Should Have — 필수 완성 후 진행 |

## 1. 회원/인증 도메인 (AUTH)

> **담당:** 팀원 A / **연결 UC:** UC-001, UC-002, UC-003

---

### FN-AUTH-01: 회원가입 🟢

| 항목 | 내용 |
|------|------|
| 연결 UC | UC-001 |
| 담당 팀원 | A |
| API | `POST /api/auth/signup` |

**기능 설명**

사용자가 이메일, 비밀번호, 닉네임을 입력하여 TicketFlow에 가입한다.

**입력값**

| 필드 | 타입 | 필수 | 검증 규칙 |
|------|------|:---:|-----------|
| email | String | ✅ | 이메일 형식 (`@` 포함), 최대 100자 |
| password | String | ✅ | 8자 이상, 영문 + 숫자 포함 |
| nickname | String | ✅ | 2~20자, 특수문자 불가 |

**처리 로직**

1. 입력값 유효성 검증 (Bean Validation)
2. 이메일 중복 조회 (`SELECT COUNT(*) FROM user WHERE email = ?`)
3. 비밀번호 BCrypt 암호화
4. User 엔티티 저장 (role = `USER`)
5. 성공 응답 반환

**응답**

| 상황 | HTTP 상태 | 응답 메시지 |
|------|-----------|-------------|
| 성공 | 201 Created | `{ "userId": 1, "email": "..." }` |
| 이메일 중복 | 409 Conflict | "이미 가입된 이메일입니다." |
| 유효성 실패 | 400 Bad Request | 필드별 에러 메시지 |

**비고**

- 관리자 계정은 직접 DB INSERT 또는 별도 시드 데이터로 생성 (회원가입 API로 생성 불가)
- 비밀번호 평문 로그 출력 금지

### FN-AUTH-02: 로그인 (JWT 발급) 🟢

| 항목 | 내용 |
|------|------|
| 연결 UC | UC-002 |
| 담당 팀원 | A |
| API | `POST /api/auth/login` |

**기능 설명**

이메일/비밀번호 인증 후 JWT Access Token을 발급한다.

**입력값**

| 필드 | 타입 | 필수 |
|------|------|:---:|
| email | String | ✅ |
| password | String | ✅ |

**처리 로직**

1. 이메일로 사용자 조회
2. BCrypt 비밀번호 검증
3. 인증 성공 시 JWT Access Token 생성
   - payload: `{ "userId": 1, "role": "USER", "iat": ..., "exp": ... }`
   - 만료: 1시간
   - 서명: HS256 + 서버 Secret Key
4. 응답 헤더 `Authorization: Bearer {token}` 또는 Body에 토큰 반환

**응답**

| 상황 | HTTP 상태 | 응답 메시지 |
|------|-----------|-------------|
| 성공 | 200 OK | `{ "accessToken": "eyJ..." }` |
| 인증 실패 | 401 Unauthorized | "이메일 또는 비밀번호가 올바르지 않습니다." |

**비고**

- 인증 실패 시 이메일 존재 여부를 구분하지 않음 (보안 원칙)
- Refresh Token은 이번 스코프에서 제외 (Out of Scope)

## 2. 이벤트 관리 도메인 (EVT)

> **담당:** 팀원 B / **연결 UC:** UC-004

### FN-EVT-01: 이벤트 등록 🟢

| 항목 | 내용 |
|------|------|
| 연결 UC | UC-004 |
| 담당 팀원 | B |
| API | `POST /api/admin/events` |
| 인증 | 🔒 JWT 필수 (ADMIN) |

**기능 설명**

관리자가 공연/스포츠 이벤트를 등록하고, 구역 및 좌석을 일괄 생성한다.

**입력값**

| 필드 | 타입 | 필수 | 검증 규칙 |
|------|------|:---:|-----------|
| title | String | ✅ | 1~200자 |
| category | Enum | ✅ | CONCERT, MUSICAL, THEATER, SPORTS |
| venueId | Long | ✅ | 존재하는 Venue ID |
| eventDate | LocalDateTime | ✅ | 현재 시간 이후 (KST 기준) |
| saleStartAt | LocalDateTime | ✅ | 예매 오픈 시각 (eventDate 이전이어야 함, KST 기준) |
| saleEndAt | LocalDateTime | ✅ | 예매 마감 시각 (saleStartAt 이후, eventDate 이전이어야 함, KST 기준) |
| roundNumber | Integer | ✅ | 1 이상 (회차 번호) |
| status | Enum | ✅ | ON_SALE, SOLD_OUT, CANCELLED, ENDED |
| description | String | ❌ | 최대 2,000자 |
| thumbnailUrl | String | ❌ | URL 형식 |
| sections | List | ✅ | 최소 1개 |
| └ sectionName | String | ✅ | 구역명 |
| └ price | Integer | ✅ | 0 이상 |
| └ totalSeats | Integer | ✅ | 1 이상 |
| └ rowCount | Integer | ✅ | 행 수 (row_name 생성 기준) |
| └ colCount | Integer | ✅ | 열 수 (col_num 생성 기준) |

**처리 로직**

1. ADMIN 권한 검증
2. venueId 존재 여부 확인
3. saleStartAt < eventDate, saleStartAt < saleEndAt < eventDate 검증 (KST 기준)
4. Event 엔티티 저장 (roundNumber, status, saleStartAt, saleEndAt 포함)
4. 각 Section 엔티티 저장
5. 각 Section의 rowCount × colCount 수만큼 Seat 엔티티 일괄 생성 (row_name = "A열"~, col_num = 1~)
6. 배치 INSERT 최적화 (`hibernate.jdbc.batch_size=50`)

**응답**

| 상황 | HTTP 상태 | 응답 |
|------|-----------|------|
| 성공 | 201 Created | `{ "eventId", "title", "totalSeats" }` |
| 권한 없음 | 403 Forbidden | "관리자 권한이 필요합니다." |
| 과거 일시 | 400 Bad Request | "이벤트 일시는 현재 이후여야 합니다." |

**비고**

- 이벤트 수정/삭제는 이번 스코프에서 제외

### FN-EVT-04: 이벤트 상태 자동 전환 (스케줄러) 🟢

| 항목 | 내용 |
|------|------|
| 연결 UC | UC-004 |
| 담당 팀원 | B |
| API | 없음 (내부 스케줄러) |
| 인증 | 없음 |

**기능 설명**

매 정각마다 `eventDate`가 현재 시각보다 이전인 이벤트를 자동으로 `ENDED` 상태로 전환한다.
관리자가 수동으로 상태를 변경하지 않아도 공연 종료 후 자동으로 상태가 갱신된다.

**처리 로직**
1. 매 정각 @Scheduled(cron = "0 0 * * * *") 트리거
2. eventDate < now() AND status IN (ON_SALE, SOLD_OUT) 조건으로 일괄 UPDATE
3. @CacheEvict(value = "event-detail", allEntries = true) — 캐시 전체 무효화
4. 전환된 건수 로그 기록

**대상 상태 전환표**

| 전 | 후 | 조건 |
|---|---|---|
| ON_SALE | ENDED | eventDate < now() |
| SOLD_OUT | ENDED | eventDate < now() |
| CANCELLED | 변경 없음 | 날짜 무관하게 유지 |

**구현 클래스**

| 클래스 | 역할 |
|---|---|
| `EventScheduler.java` | `@Scheduled` 트리거, `@CacheEvict` 처리 |
| `EventRepository.bulkUpdateEndedStatus()` | `@Modifying` JPQL 일괄 UPDATE |

**응답**

| 상황 | 처리 |
|------|------|
| 전환 대상 없음 | 로그만 기록 (0건) |
| 전환 성공 | 로그에 전환 건수 기록 |

**비고**

- `CANCELLED` 상태는 전환 대상 제외 — 취소 이력 보존 목적
- 스케줄러 실행 시 `event-detail` 캐시 전체 무효화로 오래된 상태값 캐시 방지
- 실행 주기: 매 정각 (1시간 단위) — `eventDate` 정밀도가 분 단위이므로 충분

## 3. 검색 도메인 (SRCH)

> **담당:** 팀원 B / **연결 UC:** UC-005, UC-006

---

### FN-SRCH-01: 이벤트 검색 v1 (캐시 미적용) 🟢

| 항목 | 내용 |
|------|------|
| 연결 UC | UC-005 |
| 담당 팀원 | B |
| API | `GET /api/v1/events/search` |
| 인증 | 🔓 불필요 |

**기능 설명**

키워드 + 필터 조건으로 이벤트를 검색한다. 캐시를 적용하지 않고 매 요청마다 DB를 직접 조회한다.
**v2와 동일한 인터페이스**를 유지하여 성능 비교가 가능하도록 한다.

**입력값 (Query Parameters)**

| 파라미터 | 타입 | 필수 | 설명 |
|----------|------|:---:|------|
| keyword | String | ❌ | 이벤트명 LIKE 검색 |
| category | Enum | ❌ | CONCERT, MUSICAL, THEATER, SPORTS |
| startDate | LocalDate | ❌ | 검색 시작일 |
| endDate | LocalDate | ❌ | 검색 종료일 |
| minPrice | Integer | ❌ | 최소 가격 |
| maxPrice | Integer | ❌ | 최대 가격 |
| page | Integer | ❌ | 기본값 0 |
| size | Integer | ❌ | 기본값 20 |
| sort | String | ❌ | eventDate,asc / price,asc 등 |

**처리 로직**

1. 검색어가 존재하면 Redis ZSet에 `ZINCRBY search-keywords {keyword} 1` (인기 검색어 집계)
2. QueryDSL BooleanBuilder로 동적 WHERE 조건 조합
   - keyword: `event.title LIKE %keyword%`
   - category: `event.category = ?`
   - startDate/endDate: `event.eventDate BETWEEN ? AND ?`
   - minPrice/maxPrice: 이벤트의 구역 중 하나라도 해당 가격 범위에 속하면 이벤트 노출 (m-02 대응)
     → EXISTS 서브쿼리: `EXISTS (SELECT 1 FROM section s WHERE s.event_id = e.event_id AND s.price BETWEEN ? AND ?)`
3. 페이징 + 정렬 적용
4. 결과 DTO 변환 후 반환

**응답**

| 상황 | HTTP 상태 |
|------|-----------| 
| 성공 | 200 OK (Page 응답) |
| 결과 없음 | 200 OK (빈 Page, `"검색 결과가 없습니다."` 메시지 포함) |

**비고**

- 이 API는 성능 비교 기준선(Baseline)이므로, 캐시를 절대 적용하지 않음
- 응답 시간을 로그로 기록하여 v2와 비교 가능하도록 함

**기능 설명**

FN-SRCH-01과 동일한 입력/출력 인터페이스를 가지되, **Caffeine 로컬 캐시**를 적용한다.

**캐시 설계**

| 항목 | 설정값 | 이유 |
|------|--------|------|
| 캐시명 | `event-search` | |
| 캐시 키 | `keyword:category:startDate:endDate:minPrice:maxPrice:page:size:sort` | 검색 조건 전체를 키로 사용 |
| TTL | 5분 | 이벤트 정보는 자주 변경되지 않으나, 너무 오래 캐시하면 신규 이벤트 노출 지연 |
| maximumSize | 1,000 entries | 서버 메모리 보호 (약 50MB 이하 유지 추정) |
| Eviction | LRU (Least Recently Used) | 자주 검색되는 조건이 캐시에 유지 |

**처리 로직**

1. 검색어가 존재하면 Redis ZSet ZINCRBY (인기 검색어 집계 — v1과 동일)
2. `@Cacheable(value = "event-search", key = "#cacheKey")` 어노테이션 적용
3. 캐시 히트 → DB 조회 없이 즉시 반환
4. 캐시 미스 → MySQL 조회 → 결과를 Caffeine 캐시에 저장 → 반환

**Spring 설정 (application.yml)**

```yaml
spring:
  cache:
    type: caffeine
    caffeine:
      spec: maximumSize=1000,expireAfterWrite=5m
```

**응답**

v1과 동일한 형식. 응답 헤더에 `X-Cache: HIT` 또는 `X-Cache: MISS`를 추가하여 캐시 적중 여부를 확인 가능하도록 한다.

**비고**

- v1과 v2의 응답 속도를 비교하는 테스트 코드 또는 로그 출력 필수
- 캐시 무효화: 이벤트 등록/수정 시 `@CacheEvict(value = "event-search", allEntries = true)`

```
요청 → Redis GET(캐시 키)
       ├─ HIT → 즉시 반환
       └─ MISS → MySQL 조회 → Redis SET(캐시 키, TTL 5분) → 반환
```

**캐시 설계**

| 항목 | 설정값 |
|------|--------|

| TTL | 5분 |
| 직렬화 | JSON (GenericJackson2JsonRedisSerializer) |
| 무효화 | 이벤트 등록 시 `search:*` 패턴 삭제 (`SCAN` + `DEL`, `KEYS` 사용 금지) |

**Caffeine 대비 개선점**

| 항목 | Caffeine | Redis |
|------|----------|-------|
| 캐시 범위 | 서버 1대 로컬 | 전체 서버 공유 |
| Scale-out 시 | 서버 간 캐시 불일치 | 캐시 일관성 보장 |
| 메모리 관리 | JVM 힙 사용 | 외부 Redis 메모리 |
| 네트워크 비용 | 없음 | Redis 왕복 1~2ms |

**비고**

- 필수(Caffeine)가 완성되고 테스트를 통과한 이후에 전환할 것
- v1 / v2(Caffeine) / v2(Redis) 3단계 성능 비교 보고서 작성 필수
- **전환 방식 (ADR-006):** `application.yml`의 `cache.provider` 값 변경만으로 전환 — 코드 수정 없음

  ```yaml
    # provider: redis    # Redis Cache-Aside 단계 (도전)
  ```

  ```java
  @ConditionalOnProperty(name = "cache.provider", havingValue = "caffeine")
  @Bean CacheManager caffeineCacheManager() { ... }

  @ConditionalOnProperty(name = "cache.provider", havingValue = "redis")

  ```

## 4. 좌석/Hold 도메인 (SEAT)

> **담당:** 팀원 C / **연결 UC:** UC-007

### FN-SEAT-01: 좌석 목록 조회 🟢

| 항목 | 내용                                |
|------|-----------------------------------|
| 연결 UC | UC-007 (사전 단계)                    |
| 담당 팀원 | B                                 |
| API | `GET /api/events/{eventId}/seats` |
| 인증 | 🔒 JWT 필수                         |

**기능 설명**

특정 이벤트의 구역별 좌석 상태(AVAILABLE / ON_HOLD / CONFIRMED)를 반환한다.

**처리 로직**

```
1. eventId로 Section + Seat 목록 조회
2. 각 좌석의 상태를 확인 (우선순위 순):
   a. MySQL ACTIVE_BOOKING 테이블에 해당 seat_id가 존재하면 → CONFIRMED
      (SELECT 1 FROM active_booking WHERE seat_id = ?)
   b. Redis에 hold:{eventId}:{seatId} 키가 존재하면 → ON_HOLD
   c. 그 외 → AVAILABLE
3. 구역별로 그룹핑하여 반환
```

**성능 고려사항**

좌석이 많을 경우 Redis 조회를 좌석별로 N번 수행하면 병목이 생긴다.
`MGET hold:{eventId}:1 hold:{eventId}:2 ...` 를 활용하거나, eventId 기준 hold 키를 Set으로 관리하는 방식을 고려한다.

**응답**

```json
{
  "eventId": 1,
  "sections": [
    {
      "sectionId": 1,
      "sectionName": "A구역",
      "price": 110000,
      "seats": [
        { "seatId": 1, "rowName": "A열", "colNum": 1, "status": "AVAILABLE" },
        { "seatId": 2, "rowName": "A열", "colNum": 2, "status": "ON_HOLD" },
        { "seatId": 3, "rowName": "A열", "colNum": 3, "status": "CONFIRMED" }
      ]
    }
  ]
}
```

**비고**

- ON_HOLD 상태는 Redis 존재 여부로만 판단 (DB에 ON_HOLD 상태를 별도 저장하지 않음)
- CONFIRMED 상태는 ACTIVE_BOOKING 테이블 존재 여부로 판단 (SEAT 테이블에 status 컬럼 없음)

### FN-SEAT-02: 좌석 선택 (Hold 획득) ⭐ 🟢

| 항목 | 내용 |
|------|------|
| 연결 UC | UC-007 |
| 담당 팀원 | C |
| API | `POST /api/events/{eventId}/seats/{seatId}/hold` |
| 인증 | 🔒 JWT 필수 |

**기능 설명**

사용자가 선택한 좌석을 5분간 임시 점유(Hold)한다. 동시에 동일 좌석을 선택하는 다수 사용자 중 단 1명만 성공한다.

**처리 로직**

```
1. JWT에서 userId 추출
2. [어뷰징 체크] Redis GET user-hold-count:{userId}
   └ 값 >= 4 이면 → 400 "최대 4석까지만 선택할 수 있습니다." (즉시 반환)
   ※ user-hold-count:{userId} 키 자체에 TTL 300초를 설정하므로, TTL 만료된 Hold는
      count에서 자동 제거된다. 별도 Keyspace Notification 불필요.
3. [분산락 획득] Redis SETNX lock:seat:{eventId}:{seatId} {uuid} EX 3
   └ 실패 → 409 "다른 사용자가 처리 중입니다." (Fail Fast)
4. [락 내부] ACTIVE_BOOKING 테이블에 해당 seat_id 존재 여부 확인
   └ 존재 → 락 해제 → 409 "이미 예매된 좌석입니다."
5. [락 내부] Redis EXISTS hold:{eventId}:{seatId}
   └ 이미 존재 → 락 해제 → 409 "이미 선점된 좌석입니다."
6. [락 내부] Redis SET hold:{eventId}:{seatId} {userId} EX 300
7. [락 내부] holdToken 생성 (UUID v4)
         Redis SET holdToken:{uuid} "{eventId}:{seatId}:{userId}" EX 300  ← C-01 A안
         주문 생성 시 holdToken → (eventId, seatId, userId) 역조회에 사용
8. [락 내부] Redis INCR user-hold-count:{userId}
         키가 없으면 INCR 후 EXPIRE 300 재설정 (TTL 초기화)
         키가 있으면 INCR만 수행 (TTL은 유지 — 가장 짧은 Hold 기준으로 자연 만료)
9. [락 해제] Lua Script:
   if redis.call("get", KEYS[1]) == ARGV[1] then
     return redis.call("del", KEYS[1])
   end
10. 응답: { "holdToken": "...", "expiresAt": "...(5분 후)" }
```

**응답**

| 상황 | HTTP 상태 | 에러코드 | 응답 메시지 |
|------|-----------|----------|-------------|
| Hold 성공 | 200 OK | — | `{ "holdToken", "expiresAt" }` |
| Hold 수 초과 | 400 Bad Request | `HOLD_LIMIT_EXCEEDED` | "최대 4석까지만 선택할 수 있습니다." |
| 락 획득 실패 | 409 Conflict | `SEAT_LOCK_FAILED` | "다른 사용자가 처리 중입니다." |
| 이미 선점됨 | 409 Conflict | `SEAT_ALREADY_HELD` | "이미 선점된 좌석입니다." |
| 이미 예매됨 | 409 Conflict | `SEAT_ALREADY_CONFIRMED` | "이미 예매된 좌석입니다." |

**동시성 제어 상세**

| 항목 | 설정 |
|------|------|
| 락 키 | `lock:seat:{eventId}:{seatId}` |
| 락 TTL | 3초 |
| 락 값 | UUID (본인 락만 해제하기 위한 식별자) |
| 실패 전략 | Fail Fast (재시도 없음, 티켓팅 특성상 즉시 응답이 UX에 유리) |
| 락 해제 | Lua Script로 원자적 삭제 |

**중요 비고**: Hold는 **개별 좌석 단위**로 Redis에 관리한다. 여러 좌석을 동시에 Hold한 후 결제는 BK 도메인(ORDER)에서 일괄 처리한다.

### FN-BK-01: 주문 생성 (티켓 예매 요청) ⭐ 🟢

| 항목 | 내용 |
|------|------|
| 연결 UC | UC-008 |
| 담당 팀원 | C |
| API | `POST /api/orders` |
| 인증 | 🔒 JWT 필수 |

**기능 설명**

Hold 중인 하나 이상의 좌석(최대 4석)에 쿠폰을 적용하여 주문을 생성하고, Mock PG 결제를 요청한다.

**입력값**

| 필드 | 타입 | 필수 | 설명 |
|------|------|:---:|------|
| holdTokens | List\<String\> | ✅ | Hold 토큰 목록 (최대 4개) |
| couponId | Long | ❌ | 적용할 쿠폰 ID (선택) |

**처리 로직**

```
1. JWT에서 userId 추출
2. 각 holdToken으로 Redis Hold 역조회 및 검증 (C-01 A안)
   - Redis GET holdToken:{uuid} → "{eventId}:{seatId}:{userId}" 파싱
     └ 키 없음 → 409 "점유 시간이 만료된 좌석이 있습니다." (TTL 만료 = Hold 만료)
   - Redis GET hold:{eventId}:{seatId} 값이 userId와 일치하는지 확인 (소유자 재검증)
   - 위 두 키 모두 유효해야 다음 단계 진행
3. [쿠폰 적용 시] 쿠폰 유효성 검증
   - USER_COUPON.status = ISSUED
   - 만료일 이내
   - 해당 사용자 소유
4. ORDER 생성
   - total_amount = 각 좌석의 section.price 합계
   - discount_amount = 쿠폰 할인액 (없으면 0)
   - final_amount = total_amount - discount_amount
   - status = PENDING
5. 각 좌석별 BOOKING 생성 (status=PENDING)
   - order_id = 위에서 생성한 ORDER ID
   - seat_id, event_id, user_id
   - original_price = 해당 좌석의 section.price (결제 당시 원가 스냅샷)
   ※ FAILED/PENDING 상태의 기존 BOOKING이 있어도 신규 BOOKING 생성 가능 (M-03)
      ACTIVE_BOOKING 미존재 = 좌석이 CONFIRMED 아님 = 재예매 허용
6. Hold TTL 연장: 모든 hold:{eventId}:{seatId}, holdToken:{uuid} 키에 EXPIRE 300 재실행 (+5분)
   - 주문 생성~PG 웹훅 도착 사이의 타이밍 갭 방지 (L-02 대응)
7. Mock PG 결제 요청 ({orderId, final_amount, callbackUrl})
8. 응답 반환
```

**응답**

| 상황 | HTTP 상태 | 응답 메시지 |
|------|-----------|-------------|
| 성공 | 200 OK | `{ "orderId", "status": "PENDING", "finalAmount" }` |
| Hold 만료 | 409 Conflict | "점유 시간이 만료된 좌석이 있습니다." |
| 쿠폰 무효 | 400 Bad Request | "사용할 수 없는 쿠폰입니다." |

### FN-BK-02: Mock PG 웹훅 수신 (주문 확정) ⭐ 🟢

| 항목 | 내용 |

| 담당 팀원 | C |
| API | `POST /api/mock-pg/webhook` |

**기능 설명**

Mock PG로부터 결제 결과 웹훅을 수신하여 주문을 최종 확정하거나 실패 처리한다.

**입력값 (Mock PG 웹훅 Body)**

**처리 로직 (성공 시)**

```
2. 해당 ORDER의 모든 BOOKING에서 holdToken 재검증
   - 모든 hold:{eventId}:{seatId} 키의 TTL + 소유자 재확인
   - 만료된 것이 하나라도 있으면 → 예매 실패 처리 + PG 결제 취소 요청
3. [쿠폰 적용 시] 쿠폰 사용 락 획득
   - lock:user-coupon-use:{userCouponId} 분산락 획득
   - USER_COUPON 상태 재검증 (ISSUED인지 확인)
4. DB 트랜잭션 내 일괄 처리:
   a. ORDER.status → CONFIRMED
   b. 모든 BOOKING.status → CONFIRMED
   c. 각 좌석별 ACTIVE_BOOKING INSERT
      - seat_id PK 중복 시 즉시 에러 → 트랜잭션 롤백 (DB 최종 방어선)
   d. [쿠폰 적용 시] USER_COUPON.status → USED
   e. PAYMENT 레코드 생성 (paymentKey, paidAmount)
   f. 각 BOOKING.ticket_code 생성 및 저장
      (형식: "TF-" + 연도 + 구역약자 + 좌석번호 + UUID 앞 4자리)
5. 모든 Redis Hold 관련 키 삭제
   - DEL hold:{eventId}:{seatId}         (BOOKING 수만큼 반복)
   - DEL holdToken:{uuid}                (각 holdToken 키 삭제)
   - DECR user-hold-count:{userId}       (예매 확정된 좌석 수만큼 차감, 0 이하면 DEL)
6. [쿠폰 적용 시] finally 블록에서 쿠폰 사용 락 해제
```

**처리 로직 (실패 시)**

```

2. ORDER.status → FAILED
3. 모든 BOOKING.status → FAILED
4. Hold는 유지 (사용자가 재시도 가능하도록)
5. 실패 응답 로그 기록
```

### FN-CPN-02: 선착순 쿠폰 발급 ⭐ 🟢

| 항목 | 내용 |
|------|------|
| 연결 UC | UC-011 |
| 담당 팀원 | D |
| API | `POST /api/coupons/{couponId}/issue` |
| 인증 | 🔒 JWT 필수 |

**기능 설명**

사용자가 한정 수량 쿠폰을 선착순으로 발급받는다. 동시 500명이 요청해도 정확히 설정된 수량만 발급된다.

**처리 로직**

```
1. JWT에서 userId 추출
2. [중복 발급 체크 — 락 획득 전] UserCoupon 테이블에 (userId, couponId) 존재 여부 확인
   └ 이미 존재 → 즉시 409 "이미 발급받은 쿠폰입니다." (락 불필요, 빠른 실패)
3. [잔여 수량 차감] Redis Atomic DECR + Lua Script (분산락 불필요)
   - DECR 자체가 원자적 연산이므로 추가 분산락을 거는 것은 불필요한 이중 잠금 (L-01 대응)
   - Lua Script: DECR 후 값 < 0이면 INCR로 원상복구 + 차단 (음수 방어)
   └ DECR 결과 < 0 → 409 "쿠폰이 모두 소진되었습니다."
   └ Redis 연결 자체 실패 → DB fallback (4-b 참고)
4-a. [정상 경로] UserCoupon INSERT (userId, couponId, status=ISSUED, issuedAt)
     MySQL remaining_quantity 동기화 (m-06 대응):
       UPDATE coupon SET remaining_quantity = remaining_quantity - 1 WHERE coupon_id = ?
       ※ Redis DECR 성공 직후 DB UPDATE 수행 — 매 발급마다 즉시 동기화하여 Redis 유실 시 복구 기준 유지
4-b. [DB fallback 경로 — Redis 연결 자체 실패 시]
     SELECT * FROM coupon WHERE coupon_id = ? FOR UPDATE
     remaining_quantity > 0 확인 후 차감 및 UserCoupon INSERT
     └ Redis 복구 후 remaining_quantity 기준으로 Redis 카운터 재초기화
5. 응답: "쿠폰이 발급되었습니다!"
```

**중복 발급 체크 위치 근거**:
- 락 바깥에서 먼저 체크하면 이미 받은 사용자는 락 비용 없이 즉시 차단됨
- 대규모 이벤트에서 이미 발급받은 사용자의 재시도가 많을 때 시스템 부하 감소
- (userId, couponId) UNIQUE 제약이 DB에 걸려있어 INSERT 단계에서도 최종 방어됨

**동시성 제어 상세**

| 항목 | 설정 |
|------|------|
| 수량 차감 | Redis Atomic DECR + Lua Script (원자적 — 추가 분산락 불필요) |
| DB fallback | Redis 장애 시 `SELECT ... FOR UPDATE on coupon.remaining_quantity` |
| MySQL 역할 | Source of Truth (Redis 유실 시 복구 기준) |
| 실패 전략 | Retry with backoff 최대 3회 (Redis 일시 장애 시). Redis 연결 자체 실패 시 DB fallback 경로 진입 |
| 실패 전략 선택 이유 | 쿠폰은 "받느냐 못 받느냐"가 핵심이므로, 잠깐 기다려서라도 받는 것이 UX에 유리 |

**응답**

| 상황 | HTTP 상태 | 응답 메시지 |
|------|-----------|-------------|
| 발급 성공 | 200 OK | "쿠폰이 발급되었습니다!" |
| 중복 발급 | 409 Conflict | "이미 발급받은 쿠폰입니다." |
| 수량 소진 | 409 Conflict | "쿠폰이 모두 소진되었습니다." |
| Redis 연결 실패 | 503 Service Unavailable | "현재 서비스 이용이 어렵습니다. 잠시 후 다시 시도해주세요." |
| 발급 시작 전 | 400 Bad Request | "쿠폰 발급 시간이 아닙니다." |

### FN-CHAT-02: WebSocket 연결 및 메시지 송수신 🔵

|------|------|

**STOMP 설계**

| 항목 | 경로 | 설명 |
|------|------|------|
| 연결 | `/ws-stomp` | WebSocket 핸드셰이크 엔드포인트 |
| 구독 (사용자) | `/sub/chat/room/{chatRoomId}` | 해당 채팅방 메시지 수신 |

| 전송 (사용자) | `/pub/chat/message` | 메시지 발행 |

```json
{
  "chatRoomId": 1,
  "senderId": 5,
  "senderRole": "USER",
  "content": "결제는 됐는데 예매 내역에 티켓이 없어요.",
  "sentAt": "2026-04-09T20:10:00"
}
```

**처리 로직**

1. 클라이언트가 `/ws-stomp`에 WebSocket 연결
2. STOMP CONNECT 프레임 전송 (JWT 토큰 포함)
3. 서버가 JWT 검증 후 연결 수락
4. 사용자가 `/sub/chat/room/{chatRoomId}` 구독
5. 메시지 전송 시 `/pub/chat/message`로 발행
6. 서버의 `@MessageMapping` 핸들러가 수신:
   - ChatMessage 엔티티 DB 저장
   - `/sub/chat/room/{chatRoomId}` 구독자에게 브로드캐스트

**비고**

- Spring 내장 Simple Broker 사용 (외부 메시지 브로커 불필요)
- 연결 끊김 시 STOMP reconnect 자동 재연결 (클라이언트 측 구현)
- 채팅 이력 조회 API를 별도로 제공하여 재연결 시 이전 메시지 복원

## 9. 핵심 설계 원칙 (ERD v7.0 연계)

- **서버 타임존**: **KST(UTC+9) 기준** — 모든 datetime 필드(eventDate, saleStartAt, saleEndAt, createdAt 등) 저장 및 반환 기준 (m-05 대응)
- **Hold**: 개별 좌석 단위 (Redis TTL), DB에 저장하지 않음
- **holdToken 역조회**: `holdToken:{uuid}` Redis 키에 `{eventId}:{seatId}:{userId}` 저장 (C-01 A안)
- **Hold 수 관리**: `user-hold-count:{userId}` INCR/DECR (TTL 300초) — Set 방식 대신 카운터 방식으로 TTL 자동 정리 (M-02 대응)
- **주문 단위**: 여러 좌석 + 쿠폰 적용은 **하나의 ORDER**로 처리
- **결제 단위**: ORDER (total_amount, discount_amount, final_amount 관리)
- **좌석 상태**: SEAT 테이블에 status 없음. CONFIRMED는 ACTIVE_BOOKING 존재 여부로 판단
- **ACTIVE_BOOKING**: 확정 예약만 보관. 취소 시 DELETE. 이력은 BOOKING에 남김
- **BOOKING**: ORDER에 종속 (order_id FK). original_price로 결제 당시 원가 보존
- **멀티 좌석**: 최대 4석까지 하나의 ORDER로 처리 가능
- **쿠폰 수량 동기화**: Redis DECR 성공 직후 MySQL `remaining_quantity` 즉시 차감 (m-06 대응)

## 11. FN-BK-05: 수동 예매 확정 (Admin) 🔵

| 항목 | 내용 |
|------|------|
| 연결 UC | UC-013 |
| 담당 팀원 | C (Booking 도메인) |
| API | `PATCH /api/admin/bookings/{bookingId}/confirm` |
| 인증 | 🔒 JWT 필수 (ADMIN 권한) |

**기능 설명**

PG 웹훅이 실패하거나 PENDING 상태에서 멈춘 예매를 관리자가 수동으로 확정 처리한다.
CS 채팅(UC-012)에서 고객 문의를 확인한 후 연계 처리하는 흐름이다.

**처리 로직**

```
1. ADMIN 권한 확인 (403 반환 시 즉시 종료)
2. bookingId로 BOOKING 조회 → status = PENDING 확인
   └ PENDING이 아니면 → 400 BOOKING_NOT_PENDING
3. PAYMENT 테이블에서 해당 orderId의 SUCCESS 결제 기록 확인
   └ 없으면 → 400 PAYMENT_NOT_FOUND (결제 미확인 상태에서는 확정 불가)
4. DB 트랜잭션 (SELECT ... FOR UPDATE 비관적 락):
   - SELECT * FROM booking WHERE booking_id = ? FOR UPDATE
   - SELECT * FROM `order` WHERE order_id = ? FOR UPDATE
   - SELECT * FROM active_booking WHERE seat_id = ? FOR UPDATE
   ※ 동시에 진행될 수 있는 취소 트랜잭션과의 충돌 방지 (L-04 대응)
   a. ORDER.status → CONFIRMED
   b. BOOKING.status → CONFIRMED
   c. BOOKING.ticket_code 생성 및 저장
   d. ACTIVE_BOOKING INSERT
      - seat_id PK 중복 시 → 409 SEAT_ALREADY_CONFIRMED (트랜잭션 롤백)
5. 성공 응답 반환
```

**응답**

| 상황 | HTTP 상태 | 응답 메시지 |
|------|-----------|-------------|
| 성공 | 200 OK | `{ "bookingId", "status": "CONFIRMED", "ticketCode" }` |
| 결제 기록 없음 | 400 Bad Request | "결제 기록이 확인되지 않습니다." |
| PENDING이 아님 | 400 Bad Request | "PENDING 상태의 예매만 수동 확정할 수 있습니다." |
| 좌석 이미 확정됨 | 409 Conflict | "이미 해당 좌석이 다른 예매로 확정되어 있습니다." |
| ADMIN 권한 없음 | 403 Forbidden | "관리자 권한이 필요합니다." |

## 12. FN-BK-06: 주문 상세 조회 🟢

| 항목 | 내용 |
|------|------|
| 연결 UC | UC-008, UC-010 |
| 담당 팀원 | C (Booking 도메인) |
| API | `GET /api/orders/{orderId}` |
| 인증 | 🔒 JWT 필수 |

**기능 설명**

특정 주문의 상세 정보를 조회한다. 주문 생성(PENDING) 후 결제 결과 확인(CONFIRMED/FAILED) 폴링 용도로 사용하거나,
예매 내역 상세 확인 시 사용한다.

**처리 로직**

```
1. JWT에서 userId 추출
2. orderId로 ORDER 조회 (없으면 404)
3. ORDER.userId와 JWT userId 일치 확인 (불일치 시 403)
4. ORDER에 연관된 BOOKING 목록 조회 (ticketCode 포함)
5. PAYMENT 정보 조회 (있는 경우)
6. 쿠폰 사용 정보 조회 (user_coupon_id FK로 JOIN)
7. 응답 반환
```

**응답**

| 상황 | HTTP 상태 | 응답 |
|------|-----------|------|
| 성공 | 200 OK | OrderDetail (orderId, status, bookings[], payment, couponUsed, createdAt) |
| 주문 없음 | 404 Not Found | "존재하지 않는 주문입니다." |
| 본인 주문 아님 | 403 Forbidden | "본인의 주문만 조회할 수 있습니다." |

**비고**

- 결제 후 클라이언트가 PENDING → CONFIRMED 전환을 확인하기 위한 폴링 엔드포인트로 활용 가능
- 모든 datetime 필드는 KST(UTC+9) 기준으로 반환 (m-05 대응)
